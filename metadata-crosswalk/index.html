This document is designed to promote interoperability between different accessibility metadata standards and schemas by providing guidance on ways to map between terms used in various accessibility vocabularies.

Foreword

Accessibility metadata can be provided in different ways:

  • via the metadata inside the file of the digital product,
  • via a system that maintains and produces bibliographic and supply chain metadata,
  • via a system of mapping or ‘translation’ between 1 & 2, that is from the metadata inside the product file to the corresponding metadata in the bibliographic record (which may be supplemented by additional metadata),
  • via a system of Master Data Management (MDM) containing shared production and distribution metadata.

Standards and schemas used in these different contexts may vary. This document provides information about ways to express accessibility information within the main standards used by the industry.

The crosswalk tables extends between the three standards that have different approaches and need to be mapped to each other: Schema.org, ONIX and MARC21.

The informative section provides links to standards resources. It includes additional standards beyond the ones presented in the crosswalk tables because some standards provide easy one-to-one mappings and displaying additional columns would make the tables more difficult to use.

This crosswalk is not intended to impose any kind of computer architecture on organizations but instead can serve as a guideline to compare various ways of communicating accessibility metadata.

The crosswalk is an important resource as not all parts of the accessibility metadata supply chain have access to information in all formats. Many organizations will not have access to the metadata inside digital files, which is primarily used by reading systems. Book distributors, resellers, online platforms, booksellers, librarians, educational institutions, and others will generally have access to the distributed bibliographic metadata in one or more forms, and not the metadata inside the files.

By offering a structured comparison of terms the crosswalk helps ensure that accessibility information is consistently described and discovered across platforms, making it easier for developers, content creators, and organizations to align their accessibility metadata.

Definitions are taken from the Schema.org Accessibility Properties for Discoverability Vocabulary unless otherwise noted (i.e. some are marked as being ONIX definitions).

ONIX resources

ONIX is an XML-based standard for rich book metadata, providing a consistent way for publishers, retailers and their supply chain partners to communicate a wide range of information about their products.

Within ONIX, most metadata relating to the accessibility of the product is carried in the ProductFormFeature data element. This uses ONIX codelist 196. Plus the metadata to express the access modes in the data element ProductFormFeature which uses ONIX code list 81.

To help metadata providers, EDItEUR provides two guidance documents:

  • Providing Accessibility metadata in ONIX
  • Providing accessibility metadata in ONIX (advanced)

Bidirectional mapping and round-tripping

The mappings provided in this crosswalk identify conceptual equivalence between vocabularies but are not always mathematically symmetrical or "round-trippable."

Structural differences between formats—such as strict exclusivity in one vocabulary versus permissive coexistence in another—mean that a valid mapping in one direction may be invalid in the reverse.

Example: ONIX List 196 code 09 (Inaccessible/Limited) maps conceptually to Schema.org accessibilityFeature="none". However, while ONIX code 09 allows for other accessibility features to be present simultaneously, Schema.org accessibilityFeature="none" is exclusive and prohibits other feature declarations.

In practice, ONIX records that include List 196 code 09 have few accessibility affordances, despite the option to include other features in the metadata, and can be mapped to Schema.org accessibilityFeature="none" with little loss of fidelity. Any minor accessibility features listed in the ONIX record would be left out of the Schema.org record due to the constraints Schema.org places on the term "none", which is why this transformation is not "round-trippable."

MARC21 resources

MARC 21 information

Fixed length fields (fixed fields) and variable length fields (very brief summary – fuller details)

MARC 21 fixed field = 3 digit numeric code (except the Leader); no indicators, value in each position has a specific meaning.

MARC 21 variable length field = 3 digit numeric code; each field can have 2 indicators that signify different values; each field can have one or more subfields; convention to designate subfield = $ ; if no subfield, assume $a.

More detailed information about MARC 21 field 341, and MARC 21 field 532.

Best practices and procedures for recording accessibility metadata in MARC 21 using the Schema.org Accessibility Properties for Discoverability Vocabulary (sapdv) and ONIX are still under development. Please be aware that future work may result in changes to the crosswalk, though it is expected such changes will be minimal and backwards compatible.

This crosswalk provides recommendations for MARC 21 fields appropriate for each accessibility metadata term. Cataloguing agencies can determine the correct syntax for expressing the metadata in that field.

Currently certifierReport uses MARC 21 field 856 since MARC 21 field 532 does not currently include a subfield to hold a URL. There is a proposal submitted to include this in the future.

UNIMARC resources

UNIMARC 231i (pdf file) offers a easy mapping to ONIX 196. Some references may be best described by other UNIMARC codes.

RWPM resources

Readium Web Publication Manifest (RWPM) is a JSON-based document meant to represent and distribute publications over HTTPS. It is the primary exchange format used in the Readium Architecture and serves as the main building block for OPDS 2.0.

RWPM accessibility information schema is mainly derived from the Schema.org Accessibility Properties for Discoverability Vocabulary, facilitating one to one mapping.
