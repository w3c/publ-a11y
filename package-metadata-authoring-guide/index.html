The EPUB Accessibility Metadata Guide details when and how to set accessibility metadata properties in the EPUB package document.

This guide only details usage for EPUB publications. The guidance in this document may not be relevant to other formats.

Introduction

Overview

Accessibility metadata expressed in the EPUB 3 package document [[epub-3]] is derived from three main sources: Schema.org [[schema-org]], the EPUB accessibility vocabulary [[epub-a11y]], and Dublin Core [[dcterms]].

Schema.org is a collaborative, community activity with a mission to create, maintain, and promote schemas for structured data. Although originally designed for expressing metadata on the web, it includes a wide array of metadata for describing creative works such as digital publications. In particular, it includes the following set of accessibility properties:

  • accessMode, which describes how information is encoded by default;
  • accessModeSufficient, which describes all the ways a publication can be read;
  • accessibilityFeature, which describes useful features of the content that make it more accessible;
  • accessibilityHazard, which describes both the presence and absence of potentially harmful content; and
  • accessibilitySummary, which provides additional information that cannot be captured by the other metadata.

These properties are key to the discoverability of EPUB publications and are central to the requirements of the EPUB Accessibility specification [[epub-a11y]]. They are also used to generate accessibility statements about a publication, as described in the Accessibility Metadata Display Guide for Digital Publications [[a11y-display-guide]].

To provide maximum value for readers, the metadata needs to be consistent, which is where the Schema.org Accessibility Properties for Discoverability Vocabulary [[a11y-discov-vocab]] comes in. This vocabulary defines controlled sets terms to use with each applicable schema.org accessibility property.

The accessibility properties defined in the EPUB accessibility vocabulary, on the other hand, are more limited in scope than schema.org — new properties are typically only added to express accessibility information if there are gaps in other metadata standards. Properties from this vocabulary can be identified by the a11y prefix in front of them (e.g., a11y:certifiedBy for expressing an evaluator's name)

One of the primary uses of the EPUB accessibility vocabulary is to provide additional information about conformance claims. The Dublin Core dcterms:conformsTo property [[dcterms]] is used to declare conformance to a standard such as the EPUB Accessibility [[epub-a11y]] specification, while the EPUB accessibility properties provide additional context, such as:

  • who evaluated the publication (a11y:certifiedBy);
  • any credentials help by the evaluator (a11y:certifierCredential);
  • if a full evaluation report is available (a11y:certifierReport); and
  • who to contact for more information about the accessibility of a publication (a11y:contactEmail).

The Dublin Core vocabulary also includes the dcterms:date property [[dcterms]] that is used to indicate when an evaluation was performed.

The EPUB accessibility vocabulary additionally includes properties that are not yet, or may never be, officially added to the vocabulary in the EPUB Accessibility specification [[epub-a11y]]. The most notable of these properties is a11y:exemption [[epub-a11y-exemption]], which is used to indicate that a publication fails accessibility conformance but is exempted from conformance under the laws of a jurisdiction.

The combination of all this accessibility information in the package document [[epub-3]] provides users with a better picture of whether the content is going to meet their needs. And the purpose of this guide is to provide clarity on how to apply the metadata to both reflowable and fixed-layout EPUB publications so that users get an accurate picture of the content (e.g., how to use schema.org metadata can be a challenge at first because its definitions apply to any resource on the web). The rest of this document delves into the specifics of when and how to apply accessibility metadata.

Accessibility metadata can also be expressed in supply chains through the provision of ONIX records, but that is outside of the scope of this document. For more information, refer to the Digital Publishing - Accessibility Metadata Crosswalk as well as the ONIX essentials accessibility metadata guide and advanced accessibility metadata guide.

Relationship to EPUB 2

Although the primary focus of this document is on EPUB 3 [[epub-3]], it is also expected that it can be used as a reference for adding accessibility metadata to EPUB 2 publications [[opf-201]].

Not all of the accessibility features detailed in this document apply to EPUB 2, however; in large part because it is based on the older XHTML 1.1 specification [[xhtml11]] so lacks the advanced markup features of EPUB 3. (A support table has been included that identifies which features apply to EPUB 2.)

In addition, the syntax for adding metadata to the EPUB 2 package document is not the same as adding metadata to the EPUB 3 package document. The meta tag changed significantly between versions. In particular, the name attribute was changed to property and the content attribute was replaced with text between the meta opening and closing tags. Examples in the body of the document will have to be adapted to EPUB 2's syntax.

EPUB 3 declaration of an accessibility feature:

<meta property="schema:accessibilityFeature">alternativeText</meta>

Equivalent EPUB 2 declaration:

<meta name="schema:accessibilityFeature" content="alternativeText"/>

EPUB 2 also lacks the refines attribute [[epub-3]] for connecting metadata properties. This primarily affects metadata processing when there is more than one conformance claim, as direct links between the information cannot be established. For example, which evaluator was reponsible for which conformance claim.

Content readability

Readability metadata explainer

When it comes to applying discovery metadata to an EPUB publication [[epub-3]], most of the accessibility properties are easily understood from their names — accessibilityFeature describes accessible features of the content, for example, while accessibilityHazard identifies potential physical hazards from reading the content (e.g., seizures from flashing light sequences).

Unfortunately, this clarity through naming does not extend as easily to the accessMode and accessModeSufficient properties, which often leads to initial confusion about their different roles. Their definitions also build on each other, so they can mistakenly be seen as identifying the same qualities of a publication:

  • accessMode [[schema-org]] identifies a human sensory perceptual system or cognitive faculty through which a person may process or perceive the intellectual content of a resource, not including any adaptations of the content (e.g., text alternatives for images);
  • accessModeSufficient [[schema-org]], on the other hand, identifies a list of single or combined access modes that are sufficient to understand all the intellectual content of a resource, including any adaptations.

The key difference to note is that access modes do not take into account adaptations while sufficient access modes do. Why this distinction matters will be made clear as their meanings are unpacked.

So what does it mean for a work to have an access mode?

A helpful way of thinking about an EPUB publication is to consider that it actually has two types of access mode: the access modes of the content for its primary audience and the access modes for adaptations provided to make that content more universally accessible.

To illustrate, consider a comic in which every page is a JPEG image. The publication has a visual access mode because the primary expectation is that users will have to see the pages to read the comic.

<meta property="schema:accessMode">visual</meta>

There are four primary types of access mode: textual, visual, auditory, and tactile.

But assume the publisher has also provided alternative text for each image and extended descriptions to make each page more accessible. It might seem like it should now get a textual access mode added but that would be the wrong interpretation for adaptations. Despite these accommodations to make it more accessible, the comic still only has a single visual access mode. The text alternatives are a duplication of the visual information in another form so the textual value is not also declared as an access mode.

If both visual and textual access modes were specified for this comic, it would suggest that a user requires both the ability to process visual and textual content to read the work. In fact, only one or the other is needed, and the accessMode property prioritizes the encoding for the primary audience. (The complete set of pathways by which the content can be read are captured by the sufficient access modes, to spoil the secret!)

The next question is invariably why this distinction is necessary. It is because each adaptation is a repetition of the information in another form. The adaptations do not capture new information that all readers must be able to process. They may also result in some loss of information in being adapted (e.g., a description of an image has to generalize it in some ways as it can never fully describe every aspect).

The best way to think of access modes is not strictly as accessibility metadata but as a source content description. Knowing how the content was authored provides users with a starting point to judge how it has been adapted to make it readable in other forms.

Before moving on to sufficient access modes, it might help to consider some additional cases:

  • if a novel consists only of headings and text, it has only a single textual access mode;
  • if that novel is translated to use braille Unicode characters [[unicode]], it instead has a single tactile access mode;
  • if a work of non-fiction has headings, text, and images, it has both a textual and visual access mode;
  • if the same non-fiction work were to include video with audio, it would only add an additional auditory access mode (the video component is covered by the same visual access mode as the images).

The question that arises at this point is if the access modes only describe the default encoding of the information, how is this useful for accessibility. And that is where the sufficient access modes come in.

Despite the name, sufficient access modes are not a complementary list of access modes that identify the way that the affordances are encoded. The sufficient access modes are sets of one or more access modes that describe all the ways you can read the publication from beginning to end.

The sufficient access modes answer the question of what combinations of senses and faculties can be used to read the publication in full. They account for both the source material and any adaptations that have been provided. In other words, one set of sufficient access modes is always the combination of all the access modes.

To illustrate this difference, consider the example of a textbook with images, video and audio. Its access modes would be declared as follows:

<meta property="schema:accessMode">textual</meta>
<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">auditory</meta>

One set of sufficient access modes for this publication has to be:

<meta property="schema:accessModeSufficient">textual,visual,auditory</meta>

This tells you that if you can see and hear the content you will be able to read the publication in full. Now consider if text equivalents have been included for the video and audio so that a user does not require those senses to read the publication. There is now an additional text-only sufficient access mode that should be declared for the publication:

<meta property="schema:accessModeSufficient">textual,visual,auditory</meta>
<meta property="schema:accessModeSufficient">textual</meta>

Now imagine that media overlays are used to provide a full auditory experience:

<meta property="schema:accessModeSufficient">textual,visual,auditory</meta>
<meta property="schema:accessModeSufficient">textual</meta>
<meta property="schema:accessModeSufficient">auditory</meta>

When all the adaptations are accounted for, users get a richer picture of all the ways they might be able to access the content. But this bigger picture view can also obscure the default encoding. That is why the access modes are still useful to specify but are not as generally helpful as the sufficient access modes.

In practice, when multiple access modes are specified, it is not necessary to repeat them all again as a sufficient access mode set. For accessibility purposes, only the sufficient access mode sets that consist of a single value are important (i.e., that the content is fully readable in the specified mode).

Media overlays also present a complex case because they are like combining two formats into one — an ebook and an audiobook. For more detailed information about how to set access modes and sufficient access modes when media overlays are present, refer to .

This example helps to explain another distinction between the two types of metadata. Each access mode to read the default representation of the content is listed in a separate meta tag, while each set (one or more) of sufficient access modes gets listed in its own tag, separated by commas.

The reason why this is done is because EPUB does not support lists of metadata values and does not define a method for breaking a metadata value up — doing so would lead to all kinds of problems since EPUB is an aggregator of metadata and no one rule could apply. The general rule in EPUB is that if you have more than one value, you should repeat the metadata property for each (for example, each author typically goes in its own dc:creator tag). Each access mode is one value while each set of sufficient access modes represents one value.

A final point to keep in mind when setting access modes and sufficient access modes is that they only apply to the intellectual content of the work — or, put more simply, the authored content. Publications often contain additional supplementary information that is not essential to reading the work. Not being able to see a cover image, for example, does not prevent a user from reading the work. Similarly, lists of other works by the author, author biographies, and other promotional information added by the publisher does not affect the readability. For this reason, this kind of material is ignored when determining the access modes and sufficient access modes.

Declaring access modes

Access modes are declared using the schema:accessMode property [[schema-org]], repeating the property for each applicable mode. The access modes must not be grouped together in a single tag.

<meta property="schema:accessMode">
   textual
</meta>
<meta property="schema:accessMode">
   visual
</meta>

Do not list access modes for content that does not contain information necessary to understand a publication. Most EPUB publications contain cover images, for example, but it is not necessary to see the cover image to read the publication. The same is true of publisher logos and images in the content are only for presentational purposes (i.e., have no information so have empty [[html]] [^img/alt^] attribute values). If these are the only visual content, then it is not valid to list a visual access mode. Similarly, if the only audio an EPUB publication contains is background music (e.g., for an instructional video with text captions, or as mood music while reading), listing an auditory access mode is not valid.

The access modes of adaptations are also not declared. (For more information on why not, refer to the explainer.)

The order in which the access modes are declared currently does not carry any meaning, but a useful practice is to list the access modes by their relevance to reading the content. For example, if a book is primarily text but has some images, list textual first and visual second.

EPUB Accessibility conformance

[[[epub-a11y]]] [[epub-a11y]] only recommends that the access modes of a publication be specified in the accessMode property [[a11y-discov-vocab]] as they are not as important for determining usability as the sufficient access modes.

Regardless, it is considered best practice to always specify the access modes to provide users with the most complete information about the content.

Types of access mode

The primary access modes are the ones that directly capture how the information necessary to read and understand a publication is encoded. They do not include affordances to make content accessible, such as alternative text and extended descriptions to make visual content readable. The following sections explain when to apply each of the primary access modes.

Text content

The most common access mode for EPUB publications [[epub-3]] is textual. It indicates that at least some of the information necessary to read a publication is encoded using Unicode characters [[unicode]], and is declared by setting a textual (access mode) access mode in the package document [[epub-3]].

<meta property="schema:accessMode">textual</meta>

Not all the text content in a publication is necessary to read it, so when declaring a textual access mode ensure that the text data conveys information necessary to read the authored content. Some instances that would not count toward a textual access mode include:

  • when only the table of contents in the EPUB navigation document [[epub-3]] contains text;
  • when the text is extraneous information added by the publisher, such as a list of other works by the author or a biography of the author;
  • cataloging, copyright, and other publisher information found on the title page verso;
  • an audio book that includes only the major section headings for synchronization using media overlays [[epub-3]].

Excluding this type of text content avoids giving a misleading picture of the publication. A comic, for example, might only have a page of text content for the publication info (publisher, publication date, ISBN, etc.) while the body is composed entirely of images. The text content in this case is not relevant to reading the story.

As access modes do not take into account any affordances used to make non-textual content accessible (such as alternative text and descriptions for images or transcripts for audio), do not declare a textual access mode if these are the only text content (e.g., for a manga or comics).

A textual access mode is also not declared if the only text content is encoded as images. For example, headings represented as images in order to preserve unique styling and images of text in the panels of comics are not considered in determining if there is a textual access mode.

If the text data is preserved separately from the image data — for example, by using the [[svg2]] title or desc elements — then a textual access mode would be declared as such data is accessible to assistive technologies and can sometimes have its appearance modified by reading systems [[epub-3]].

Visual content

It is common for EPUB publications [[epub-3]] to contain some visual content. Some publications, such as comics and manga, will even be entirely image based. But visual does not mean the information is only in static images. Visual content also encompasses videos as well dynamic content such as interactive images or games drawn on the [[html]] [^canvas^] element.

When at least some of the information necessary to read a publication is encoded in visual form, a visual (access mode) access mode is declared in the package document [[epub-3]].

<meta property="schema:accessMode">visual</meta>

Not all visual content in a publication carries information necessary to its understanding. Examples of when a visual access mode would not be set include:

  • for the cover page;
  • for visual content that is purely presentational (e.g., decorative images and games only meant to engage students without adding any new information);
  • for visual content that is extraneous information added by the publisher (e.g., corporate logos, author headshot, etc.)

Not setting a visual access mode for these kinds of cases avoids users mistakenly believing that they have to be able to see the authored content in order to read and understand it.

Visual content indicators

The visual content indicators are a special class of values that extend the visual access mode. As a supplementary piece of information, their use is only advisory and they must always be paired with a visual access mode declaration.

The visual content indicators go beyond the basic need to visually perceive the content to provide additional information about the type of content that is encoded visually. For example, although support for MathML [[mathml3]] has improved in recent years, it is still common to find images used for math equations to ensure their rendering. Being able to inform users who are blind that a publication has images of math lets them know that they will have to rely on alternative text and extended descriptions instead of being able to inspect the equations in the kind of detail that MathML allows.

The visual content indicators are also useful for visual readers as they indicate that which types of content cannot be resized (only zoomed) or have its appearance changed to represent it more accessibly.

Like with the visual access mode, the visual content indicators are not exclusive to image formats. If a video has information that requires color perception to understand, for example, a color dependence indicator could be set. Or, if an interactive chart is drawn on an [[html]] [^canvas^] element, the chart on visual could be set.

Whether or not the content identified by a visual indicator is made accessible in another form (e.g., through the provision of alternative text) is not considered when setting these values. Whether the content is fully readable in a non-visual form is captured by the sufficient access modes.

More than one visual content indicator can apply to a piece of visual content. A publication with an image of a chart that uses color to represent the data, for example, would include both the chart on visual and color dependence indicators. When declaring visual content indicators, make sure to check the publication for all that apply.

The following list describes the types of visual content indicators and how to apply them.

Charts

The chartOnVisual indicator is used to declare that there are visual representations of data, such as pie charts, bar charts, scatter plots, and line charts.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">chartOnVisual</meta>

For visual representations of concepts, such as in flowcharts, refer to the diagramOnVisual indicator.

Chemistry

The chemOnVisual indicator is used to declare that chemical equations and formulas are represented only in visual form.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">chemOnVisual</meta>

The chemOnVisual indicator is typically used to indicate that chemical content is represented using static image formats like JPEGs and PNGs. It could also apply if a publication includes a video that, for example, shows chemical formulas and equations on screen that the reader needs to understand but that are not described in an audio track.

If the chemical content is decorative, then this value is not declared. For example, a cartoon might show a character with a swirl of superficial equations over their head to indicate confusion. As the actual equations are irrelevant to understanding the content, this would not represent a case of visual chemical content.

MathML and LaTeX are sometimes used to represent chemical content in a publication. In these cases, although the final equation is drawn visually, the fact that the drawing is based on structured markup or formatting means it is not purely visual; assistive technologies can help users step through the underlying structure. The use of MathML and LaTeX are instead represented as accessibility features.

Color dependence

The colorDependent indicator is used to declare that visual content requires the ability to perceive color in order to understand the information being conveyed.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">colorDependent</meta>

An example of color dependence would be a bar chart where each bar is a different color and a color-coded key is the only way to determine the meaning of each color. A user who cannot differentiate the colors could not match the key values to each bar.

This value does not apply if there are secondary indicators that allow users to distinguish the information. For example, if the bar chart in the previous example also had lines running in different directions to differentiate each bar, then a user could use those patterns to match the bars to the key. Similarly, if keywords are placed within each bar instead of using a separate color key, there would be no chance for confusion.

The color dependence indicator is not intended to capture issues with CSS styling, such as hyperlinks that might not be perceivable, or student exercises coded by border or background color. If a book fails accessibility conformance for these kinds of color problems, it should be noted in the accessibility summary.

Diagrams

The diagramOnVisual indicator is used to declare that there are visual representations of concepts, such as Venn diagrams, flowcharts, mind maps, and family trees.

<meta property="schema:accessMode">visual</meta>
      <meta property="schema:accessMode">diagramOnVisual</meta>

For visual representations of data, such as in pie and bar charts, refer to the chartOnVisual indicator.

Math

The mathOnVisual indicator is used to declare that math equations and formulas are represented in visual form.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">mathOnVisual</meta>

The mathOnVisual indicator is typically used to indicate that chemical content is represented using static image formats like JPEGs and PNGs. It could also apply if a publication includes a video that, for example, shows mathematical formulas and equations on screen that the reader needs to understand but that are not described in an audio track.

If the math content is decorative, then this value is not declared. For example, a decorative image at the start of a chapter might depict equations that will be covered but the actual equations in the body use MathML markup.

MathML and LaTeX are also used to represent mathematical content in a publication. In these cases, although the content is drawn visually, the fact that the equation or formula is based on structured markup or formatting means it is not purely visual; assistive technologies can help users step through the underlying structure. The use of MathML and LaTeX are instead represented as accessibility features.

Music

The musicOnVisual indicator is used to declare that musical notation is represented visually.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">musicOnVisual</meta>

Unlike math and chemical content, there is no support for rendering a music markup language such as MusicXML. Visual representation, often in image form, is the most common way to add music notation. (Text tablature notation can be added to EPUBs for stringed instruments, but as unstructured text it is also not easily accessible to users who cannot visually read the notation.)

Images are not the only way that music can be visually encoded. An instructional video in a music book could show musical notation to allow the user to play along with a recorded sample. This would also necessitate declaring a visual music indicator.

Text

The textOnVisual indicator is used to declare that there are images of text.

<meta property="schema:accessMode">visual</meta>
<meta property="schema:accessMode">textOnVisual</meta>

There are many reasons why images of text are included in EPUB publications. Appearance is one, as having to faithfully represent the appearance of the text in a print edition because font and CSS support cannot reliably replicate an effect is sometimes non-negotiable for a publisher. Or an original document, like a newspaper clipping or historical document, is reproduced exactly as it was printed. It also includes the text encoded in the images of comics and manga, children's books, cookbooks, and other image-based fixed layout publications.

As with the other visual indicators, the representation of text in visual form also encompasses video content. A silent instructional video, for example, might include text directives directly in the video.

In all these cases, it would be appropriate to list textOnVisual as an access mode. Only if the text in the visual content is in a decorative image or not directly relevant to understanding the text is there no need to specify the textOnVisual indicator.

For example, the text on signs and billboards in the background of a photograph do not represent text on visual imagery if they are incidental to what the image is conveying. Similarly, a publisher's name in their logo should not be declared as text on visual content.

The use of SVG is a special case when semantic elements are used to add the text to the image. When this is done in reflowable EPUBs, the text on visual indicator can be omitted as there is the possibility to access and change the appearance of the text. But it would not be true in a fixed layout EPUB as reading systems [[epub-3]] restrict control over the appearance of the pages.

If SVG's text elements are not used to display the text, then textOnVisual is specified since the text is not separate from the drawn image.

Auditory content

Auditory content includes any content that requires a user to be able to hear it in order to understand it. The most common forms of auditory content are standalone audio clips embedded in a publication and any video that is accompanied by an auditory track, but sounds can be important in other ways. If an interactive graphic drawn on the [[html]] [^canvas^] element includes auditory cues or announcements, for example, or if auditory cues are the only indication to users that time is running out on an interaction, auditory perception is required.

When at least some of the information necessary to read a publication is encoded in auditory form, an auditory (access mode) access mode is declared in the package document [[epub-3]].

<meta property="schema:accessMode">auditory</meta>

Not all auditory content in a publication carries information necessary to its understanding. Examples of when an auditory access mode would not be set include:

  • for mood music that plays in the background while reading;
  • for background music in an instructional video when all the information is conveyed visually;
  • for the background sound to an interactive game when it is solely for entertainment value.

Audio synchronized with the text of an EPUB publication also represents a unique case as it is often considered an adaptation. This case is discussed in detail in the next section.

Not setting an auditory access mode for these kinds of cases avoids users mistakenly believing that they have to be able to hear the authored content in order to read and understand it.

Synchronized text and audio

A special case for audio in EPUB 3 involves text and audio synchronization using the media overlays feature [[epub-3]]. Setting the correct access modes and sufficient access modes for EPUB 3 publications that contain synchronized text-audio playback requires evaluating whether playback is essential to reading the publication or whether it is included as an optional feature.

One of the more common uses of media overlays, particularly in mainstream publishing, is to provide synchronized playback of the full text of a publication with full audio narration. These types of publications are commonly referred to as "read aloud" books, as the user chooses whether or not to turn on the narration (unlike traditional audiobooks where only the audio is available).

In this case, because the audio playback is an extra feature, and is not required for reading the publication, it is not listed as an auditory access mode. Having full audio playback capability in this case instead means that an auditory sufficient access mode needs to be declared — the user can listen to the complete publication as an alternative to reading the text. (The presence of text and audio synchronization is also declared using the synchronizedAudioText accessibility feature.)

<meta property="schema:accessMode">textual</meta>
<meta property="schema:accessModeSufficient">textual</meta>
<meta property="schema:accessModeSufficient">auditory</meta>
<meta property="schema:accessibilityFeature">synchronizedAudioText</meta>

For metadata purposes, an EPUB can only be one type of publication. It cannot be separately declared that a publication is both an ebook and an audiobook — EPUB's metadata is too limited in its expressivity to handle this kind of dual role.

That is why when media overlays provide a fully synchronized audio experience one presentation has to be defined as primary and the other an affordance. Because EPUB is primarily considered an ebook format, the audiobook-like experience is treated as an affordance for the metadata to make sense. The ability for users to discover that the publication is readable entirely by audio is not lost because of this choice; it is just transferred to the sufficient access modes.

Another use for media overlays — more typical among accessible republishers such as libraries that serve blind and low vision readers — is to provide full audio synchronized to the major headings of a publication (typically referred to as "structured audio"). These types of publications are more like traditional audiobooks, as all the information in the work is typically available in auditory form. The minimal text does not allow meaningful visual reading or text-to-speech playback; it is only to provide structured navigation of the audio.

In this case, being able to hear the audio is essential to being able to read the publication, so an auditory access mode is declared.

In a reverse of the first case discussed, text-audio synchronization is not identified as a feature of the publication since the amount of text provided is trivial.

Likewise, since the text in the publication only provides heading navigation, a textual access mode is not listed. The user does not need to, and is not expected to, visually read this text.

<meta property="schema:accessMode">auditory</meta>
<meta property="schema:accessModeSufficient">auditory</meta>

Tactile content

Tactile content is less common in mainstream EPUB publications [[epub-3]] but it is sometimes found in content produced specially for users who are blind, and in particular is central to the eBraille format which is based on EPUB 3.

The tactile access mode partly overlaps with both textual and visual in what it represents, but the difference is in how. Using braille Unicode characters (see Braille Patterns [[unicode]]) to represent the text content is a direct corollary to a textual access mode, while tactile graphics and objects are how visual imagery get represented.

A tactile access mode is never declared simply because it is possible to convert standard text content into braille on the fly. Users seeking to find content that can be converted in this way will look for a single textual sufficient access mode.

When at least some of the information necessary to read a publication is encoded in tactile form, a tactile (access mode) access mode is declared in the package document [[epub-3]].

<meta property="schema:accessMode">tactile</meta>

The same as with the textual and visual content it represents, not all tactile content in a publication may carry information necessary to its understanding. The inclusion of inessential textual and visual content in a fully tactile edition, however, is generally quite rare. But a mainstream publication could, for example, include links to embossable tactile graphics or printable 3D objects. As this type of content would typically be added as an affordance, or at least a supplementary aid, a tactile access mode would not be declared.

In the case of linked content, a tactile graphic or tactile object accessibility feature would be declared instead.

Sufficient access modes

For an introduction to sufficient access modes and how they differ from access modes, refer to the readability metadata explainer.

Declaring sufficient access modes

Sufficient access modes are declared in the schema:accessModeSufficient property [[schema-org]].

<meta property="schema:accessModeSufficient">
   textual
</meta>

Unlike the other schema.org metadata expressed in the package document [[epub-3]], sufficient access modes can consist of one or more values separated by a comma and optional space. Because sufficient access modes are telling users the different ways that the content can be read, each set is expressed in a separate tag (unlike access modes where each applicable value goes in its own tag).

The most common reason for having multiple schema:accessModeSufficient declarations is to capture both the default nature of a publication as well as the alternative reading pathways available. For example, that a book with textual and visual content is also fully readable as text.

The first declaration indicates that the content is readable if text and visual content can be processed while the second indicates that there is a purely textual means of reading the content (i.e., text alternatives are available for the visual content). 

<meta property="schema:accessModeSufficient">
   textual,visual
</meta>
<meta property="schema:accessModeSufficient">
   textual
</meta>
EPUB Accessibility conformance

[[[epub-a11y]]] [[epub-a11y]] requires that the sufficient access modes of a publication be specified in the accessModeSufficient property [[a11y-discov-vocab]].

The single sufficient access modes, when applicable, are the most important to list. Combined sets can also be listed but are not as critical.

Single reading modes

Although sufficient access mode are expressed as sets consisting of one or more values, the most strongly recommended sets to list are the ones that consist of only a single value. The reason why these are so important is because they tell users that all the content can be read in the specified method.

Someone who is blind, for example, might want to know that all the content is available in text form so that they can use their reading system's [[epub-3]] built-in read aloud functionality to read the book. Or they might want to know that all the content is available as pre-recorded audio for enhanced playback.

Combinations are, of course, still of value to list, but when multiple modes are required, it is often the case that one or more will be problematic. A book that has a sufficient access mode set that indicates textual and auditory content, for example, will be usable by a user who is blind but not by a user who is deaf.

The possible single sufficient access mode values are described in the following sections.

Textual

Setting the textual (sufficient access mode) value indicates that information necessary to read a publication is available in text form.

Unlike a textual primary access mode, the inclusion of text alternatives such as alternative text, extended descriptions, closed captions, and transcripts are considered when determining whether there is a textual sufficient access mode.

When textual is the only sufficient access mode set, it means that all the information necessary to read the publication is available in textual form, so it is generally interpreted as indicating that the publication is screen-reader and dynamic braille display friendly.

<meta property="schema:accessModeSufficient">
   textual
</meta>

A single textual sufficient access mode can be set for EPUB publications [[epub-3]] that conform to Level A [[wcag2]] or higher, for example, as there must be textual alternatives for non-text content to meet these thresholds.

Visual

Setting the visual (sufficient access mode) value indicates that information necessary to read a publication is available in visual form, such as in images and video.

When visual is the only sufficient access mode set, it means that all the information necessary to read the publication is available in visual form. It is most often applied to works like comics and manga.

<meta property="schema:accessModeSufficient">
   visual
</meta>

A single visual sufficient access mode is typically the least accessible single sufficient access mode that can be set, as it indicates that only sighted readers can access the information of the publication. It does not, by itself, mean the publication is not accessible, but there would need to be another single sufficient access mode declaration to indicate the publication is readable by non-visual users.

Auditory

Setting the auditory (sufficient access mode) value indicates that information necessary to read a publication is available as prerecorded audio (whether prerecorded human speech or prerecorded synthetic speech).

When auditory is the only sufficient access mode set, it means that all the information necessary to read the publication is available as prerecorded audio.

<meta property="schema:accessModeSufficient">
   auditory
</meta>

A single auditory sufficient access mode typically indicates that media overlays have been used to provide audio for all the text and visual content, but it could also set if, for example, a book of poetry also contained (non-synchronized) recordings for each poem. More information about how to set sufficient access modes when media overlays are used is available in .

When media overlays are present, the "auditory" value should not be added to all the possible sufficient access modes just because it is possible to turn on synchronized text-audio playback.

For example, a publication with media overlays that has text and images (with text alternatives sufficient to understand their content) would only declare the following sufficient access modes:

<meta property="schema:accessModeSufficient">textual</meta>
<meta property="schema:accessModeSufficient">auditory</meta>
<meta property="schema:accessModeSufficient">textual,visual</meta>

It would not include declarations for "textual,auditory" or "textual,visual,auditory".

(Note that the final "textual,visual" set captures the two access modes needed to read the default content — textual for the text content and visual for the images.)

An auditory sufficient access mode is not set for textual content that the user might be able to get rendered as synthetic speech via their reading system [[epub-3]]. The auditory content must come prerecorded with the publication.

Tactile

Setting the tactile (sufficient access mode) value indicates that information necessary to read a publication is available in tactile form. It indicates some mix of braille Unicode characters [[unicode]], tactile graphics, and tactile objects are included in the publication.

Unlike a tactile primary access mode, providing links to tactile graphics and objects does allow a tactile sufficient access mode to be set. Such linked content, however, rarely correlates with a publication being fully readable in tactile form; see the next section on combined reading modes.

When tactile is the only sufficient access mode set, it means that all the information necessary to read the publication is available in tactile form.

<meta property="schema:accessModeSufficient">
   tactile
</meta>

A single tactile sufficient access mode is the least common single sufficient access mode, at least for mainstream publishing. It is typically associated with a braille publication produced using the eBraille format.

Combined reading modes

Although the single sufficient access modes are considered the most important to set, it is also helpful to give users a complete picture of all the ways the content can be read, especially if the primary access modes are not set.

Publications that have more than one primary access mode always have a sufficient access mode set that is the combination of all those primary modes. For example, a book with text and images has textual and visual primary access modes. Since those two values tell the user that they have to be able to read text content and perceive visual content, the content has a sufficient reading mode that combines the textual and visual values.

<meta property="schema:accessMode">
   textual
</meta>
<meta property="schema:accessMode">
   visual
</meta>
<meta property="schema:accessModeSufficient">
   textual,visual
</meta>

Because sufficient access modes account for affordances, the possible sets of sufficient access modes are not always strictly combinations of the primary modes (even though there is always one set that equals all the primary modes).

In this example, a comic provides alternative text but omits needed extended descriptions to make the panels fully accessible. It has a sufficient access mode set that includes visual and textual because the text only makes the content partially readable (visual perception is still necessary).

<meta property="schema:accessMode">
   visual
</meta>
<meta property="schema:accessModeSufficient">
   visual
</meta>
<meta property="schema:accessModeSufficient">
   visual,textual
</meta>

The combinations of possible sufficient access modes grows as there are more primary access modes and affordances to make them accessible. This is why it is not always helpful to users to have all the possibilities listed. It can make the metadata more confusing to understand.

Accessibility features

Overview

Setting features

Identifying the accessibility features and adaptations included in an EPUB publication [[epub-3]] allows users to determine whether the content is usable at a more fine-grained level than the access modes do.

For example, a math textbook might have a textual access mode, but that alone does not indicate whether MathML markup is available. Whether a visual work only provides alternative text or whether it includes extended descriptions is also important to know when gauging its usability.

Accessibility features are identified in the schema:accessibilityFeature property [[schema-org]]. The property is repeated for each feature — do not group all the features into one tag.

<meta property="schema:accessibilityFeature">
   MathML
</meta>
<meta property="schema:accessibilityFeature">
   alternativeText
</meta>

The EPUB format requires that some accessibility features will always be present (e.g., a table of contents). Do not exclude these features from the accessibility metadata, as users typically are not aware what features are built into a format. Failing to include entries will reduce the discoverability of the publication when users search for specific features.

EPUB Accessibility conformance

[[[epub-a11y]]] [[epub-a11y]] requires that the accessibility features of an EPUB publication [[epub-3]] be specified in the accessibilityFeature property [[a11y-discov-vocab]]. This means that at least one accessibility feature must be declared in the package document [[epub-3]] metadata in order for an EPUB publication to claim conformance.

Although the vocabulary for the accessibilityFeature property contains the values "none" and "unknown", these terms cannot be used to meet the reporting requirements for the property. At least one feature that is not one of these values is required to claim conformance to [[[epub-a11y]]].

When to claim features

One of the most common questions about accessibility features is not what they refer to but when they can be claimed for an EPUB publication [[epub-3]]. Features may not be applicable in every situation, for example, such as extended descriptions when there is no visual content. Or the features may not be provided in every situation where they might be needed.

Ideally, an accessibility feature will only be claimed when it covers all applicable cases, as this will make the content optimally accessible. But limiting reporting to this standard would mean that users will have difficulty determining if content that may not be fully conforming to accessibility standards may still be usable to them.

Consequently, a measure of common sense is necessary when deciding whether to declare a feature that is not applied universally. The goal of this metadata is to help users determine the suitability of the content, not to make publishers feel better about content that they know they have not made fully accessible.

If only one out of a hundred images that needs extended descriptions has one, for example, claiming that the publication provides this feature is clearly stretching the truth. But if half the images are described, there is some value to users.

The key when not providing full coverage is to detail why not in the accessibility summary so that users are aware of the limitations up front. The following information is especially useful to provide when making claims against incomplete coverage:

  • why there are coverage gaps and if an update is planned to fix them;
  • a rough estimate of how complete the coverage is; and
  • if possible, which sections or chapters have complete coverage and which do not.

When features are not known

There may be times when the presence of accessibility features is not known but a statement is still needed in the package document [[epub-3]] metadata. This usually happens if a placeholder is needed until an accessibility evaluation is carried out.

To indicate that the presence of accessibility features is not known, the unknown (feature) accessibility feature is declared.

<meta property="schema:accessibilityFeature">unknown</meta>

The unknown value is not intended for permanent use in an EPUB. If a publication is distributed with the value because the publisher cannot wait on an accessibility review to complete, it is expected that they will release an updated publication with new metadata once the evaluation is complete.

The unknown value must not be set if any other accessibilityFeature value is also declared. Declaring unknown acts as a temporary placeholder, explicitly stating that an accessibility audit has not yet been performed. It distinguishes the content from a publication that has been audited and confirmed to have no features.

When there are no features

It is first worth noting that it is rare that there are no accessibility features in an EPUB publication [[epub-3]], even if the publication as a whole does not meet minimum accessibility conformance standards. Except for some edge cases, all EPUBs can typically claim to provide a table of contents, for example, since this is a required component of the standard. It is also most often the case that reflowable EPUB publications can claim a reading order.

But in the case where an EPUB publication provides no accessibility features, the lack of features is indicated in the package document [[epub-3]] metadata using the none (feature) value.

<meta property="schema:accessibilityFeature">none</meta>

This value must not be set if any other accessibilityFeature value is declared. Declaring the none value provides an explicit and unambiguous statement that no specific accessibility features have been included, which aids in content discovery. This is different from a publication that simply omits all accessibility metadata.

Organization of this section

Accessibility features are addressed in this document based on common use patterns rather than as individual values in isolation.

All the features that are used to address the accessibility of audio and video are discussed together, for example, to better explain the complimentary uses of the features. This also allows features to be discussed that have mutually-exclusive values, such as whether full or partial ruby annotations are provided.

A navigation menu is included at the start of each section to help locate which features are addressed in it. In addition, the index of terms provides links to all the features and other properties defined in this document.

Aural rendering

In this section:

  • synchronizedAudioText
  • ttsMarkup

EPUB 3 publications allow the aural rendering of text content through the use of text-to-speech (TTS) playback (commonly referred to as "read aloud" capability) and media overlays [[epub-3]]. Both are capable of providing audio playback of the text synchronized with highlighting of the associated text.

Text-to-speech engines, however, often fail to provide accurate readings of the text. Heteronyms, for example, where two words are spelled the same but have different pronunciations, often result in the wrong word being pronounced. Complex words, such as chemical names and technical jargon, also result in illegible playback, as do fictional words and names created by an author (for example, in works of science fiction). TTS playback also suffers from an inability to indicate emphasis or to control the pacing and prosody of the narration.

It is possible to provide enhancements in the markup of EPUB publications [[epub-3]] to help TTS engines render text accurately. EPUB 3's SSML attributes [[epub-tts]], for example, were intended to allow the inlining of pronunciation information, while PLS lexicons [[pronunciation-lexicon]] were designed to provide a reusable library of pronunciations. CSS Speech [[css-speech]] properties address the issues text-to-speech playback quality (for example, the voice to use, whether to spell out words, and when to provide emphasis).

Unfortunately, at this time none of these technologies have been proven to work at any kind of scale in EPUB reading systems [[epub-3]]. Despite this shortcoming, if a publisher makes the effort to enhance TTS playback for a specific regional use case, they can indicate this by adding the ttsMarkup feature to the package document [[epub-3]].

<meta property="schema:accessibilityFeature">ttsMarkup</meta>

It is not required that all the TTS enhancements be used in order to specify this feature. If only one applies to a publication, or is known to work in reading systems, the feature can still be specified. It can also be used if new TTS enhancements are developed beyond those listed in this section.

It is only recommended to set this feature if the TTS enhancements substantively cover the rendering issues that are likely to arise — recognizing that different TTS engines have different capabilities and authors cannot anticipate every misreading they might make.

Setting the feature does not specify which TTS enhancement feature(s) have been used, so it is also good practice to indicate why the feature is being set in the accessibility summary.

Media overlays differ from text-to-speech playback in that they provide prerecorded audio synchronized with the text. The synchronization, and granularity of playback synchronization, is set by the creator of the EPUB publication through markup files that conform to a profile of the SMIL [[smil]] standard.

When including media overlays, the synchronizedAudioText feature is specified in the package document.

<meta property="schema:accessibilityFeature">synchronizedAudioText</meta>

The synchronizedAudioText feature is not declared for publications that only contain structured audio — where the only text is the major headings of the work to facilitate basic navigation of the audio. The text in these cases does not sufficiently match the audio and is not typically expected to be displayed.

The audio provided with media overlays is typically human narrated but some producers also use text-to-speech engines to create the audio. For example, prerecorded text-to-speech narration is often used when providing audio-synchronized playback of backmatter, which is tedious for humans to read.

The use of the word "text" in the feature name should not be interpreted solely as the audio having to be synchronized with Unicode text character content. It is intended to be understood more like the text element that media overlays use to synchronize playback. The text element can also be used to refer to any type of content, spans of Unicode text, images with text, images of text, images with no direct text in them but that have alt text and/or (possibly hidden) extended descriptions, as well as other possibilities. In other words, only focus on whether media overlays are provided, not on what each text element refers to, when setting this feature.

Media overlays also makes it possible to synchronize dynamic text-to-speech rendering with text highlighting, although this is not commonly done or supported by reading systems.

Whether all or only some of a publication's content is covered by synchronized text and audio playback cannot be determined by the presence of this feature alone. Completeness of coverage can be calculated by inspecting the access modes and sufficient access modes that have been set. For more information, refer to .

The use of the TTS enhancements and media overlays is not mutually exclusive. TTS markup enhancements could be provided, for example, to improve the quality of the media overlays dynamic TTS generation feature. Some non-visual readers also prefer to use a reading system's read aloud capabilities even when media overlays are available as it allows them to read faster.

Chemistry

In this section:

  • latex-chemistry
  • MathML-chemistry

Providing accessible markup for chemical formulas and equations allows users to inspect the equations themselves rather than rely on images of the content with alternative text and descriptions. Even when care is taken in the writing of descriptions, ambiguities can arise because human language is not always as precise as the equations themselves.

Accessible markup also provides a robust, accessible alternative to images, as the notation can be resized, searched, copied, and interpreted correctly by screen readers.

Although EPUB 3, and web browsers more broadly, do not currently support ChemML, there are still two markup syntaxes that can be used to add accessible chemical formulas and equations.

The first of these, despite its name, is MathML [[mathml3]]. MathML provides a structured, semantic representation that can be rendered directly by modern browsers and assistive technologies. MathML markup can be added directly to XHTML content documents [[epub-3]]. Just as it is able to represent mathematical equations, its syntax is also adaptable to representing chemical equations and formulas.

When an EPUB publication contains chemical formulas and equations marked up as MathML, the MathML-chemistry feature is set in the package document [[epub-3]].

<meta property="schema:accessibilityFeature">MathML-chemistry</meta>

It is not required that every chemical equation or formula be represented as MathML in order to set this value, but the coverage should be substantive. Judgement will always be involved in determining whether MathML is necessary for every equation. Some simple equations may not benefit from the additional markup overhead.

The other way to represent chemical equations and formulas is using the LaTeX typesetting system. LaTeX rendering is not natively supported in reading systems [[epub-3]], so it is often only used by publishers for preparing print manuscripts and transformed into another form, like images, for digital publications. But it is possible to embed LaTeX-formatted equations and use a JavaScript library like MathJAX to translate them for visual reading (the caveat, of course, is that not all reading systems support scripting).

Another possible use for LaTeX-formatted equations is as the alternative text for images, although this can lead to poor pronunciation by assistive technologies.

One reason for including the raw LaTeX is that it offers users the opportunity to copy the equations and use them in an application that allows them to read the syntax. But whether users are able to do this depends on whether they know how to unzip an EPUB and locate the equations, and whether digital rights management schemes even allow them access into the EPUB.

For these reasons, it is not common to find LaTeX-formatted chemical equations and formulas in EPUBs, but if the syntax is provided the latex-chemistry feature is set in the package document.

<meta property="schema:accessibilityFeature">latex-chemistry</meta>

Similar to setting MathML-chemistry, it is not required that every chemical equation or formula be represented as LaTeX in order to set this value, but the coverage should be substantive for all complex equations.

Although in theory it is possible for an EPUB publication to contain chemical equations and formulas encoded both in MathML [[mathml3]] and LaTeX, such dual encoding is not found in practice. Publishers use one or the other syntax.

Display control

In this section:

  • displayTransformability

The displayTransformability accessibility feature value is used to indicate that the user can modify the presentation of the text content of an EPUB publication [[epub-3]] without negatively affecting the readability.

<meta property="schema:accessibilityFeature">displayTransformability</meta>

When to set the displayTransformability value depends on the characteristics of the language a publication is written in. In general, the following characteristics serve as a basis for what must be modifiable:

  • the text foreground and background colors;
  • the font family and size;
  • the paragraph spacing;
  • the line height;
  • the letter and word spacing;
  • the text alignment.

Making sure content is transformable on these characteristics will be sufficient for many languages, but not all of these characteristics will apply to all languages. And depending on the language, it can be as important, or more so, to ensure that other requirements are modifiable, such as:

  • the writing direction (horizontal or vertical);
  • the display of ruby (both visibility and text color);
  • the inclusion of space characters as word separators.

Providing guidelines tailored to every language is outside the scope of this guide so publishers will need to take into consideration local accessibility needs when determining if the content is sufficiently transformable. One measure of conformance is to match the transformability to common display control properties provided by reading systems [[epub-3]] tailored to the local language.

To claim this feature, text must not be represented as images unless it falls under one of the exceptions specified in success criterion 1.4.5 [[wcag2]] or the text is incidental to the purpose of the image (e.g., text on a billboard or street sign in the background of an image).

In general, reflowable EPUB publications will meet display transformability requirements so long as none of the content is constrained visually by its bounding box. This typically only happens if overflow is clipped using CSS to prevent scroll bars from appearing. An example would be if a publisher adds a sidebar with a fixed height and width and clips any content that exceeds these dimensions. If the text is sized to neatly fit in this box only at the publisher's preferred setting, increasing the font size or spacing would cause text to progressively disappear.

This feature is not applicable to fixed layout EPUBs because reading systems do not allow users to modify the display. Reading systems must support the accessibility of a feature before it can be claimed.

Enhanced display

In this section:

  • highContrastDisplay
  • largePrint

There are two values for indicating that the display of an EPUB publication [[epub-3]] has been enhanced by the publisher.

The first of these is the highContrastDisplay feature, which is used to state that there is a higher-than-normal contrast between text and its background. Providing a high level of contrast increases the legibility of the text for users with low vision or color vision deficiencies.

<meta property="schema:accessibilityFeature">highContrastDisplay</meta>

This feature can be set when text and background color combinations meet or exceed the 7:1 contrast ratio for normal-sized text. It is the equivalent of meeting [[wcag2]] success criterion 1.4.6.

Note that to meet the requirements for setting this feature that images of text also have to meet the contrast requirements. The only exceptions are defined in success criterion 1.4.6, with the most typical exception for digital publications being for publisher and imprint logos.

The schema.org vocabulary does not define feature for meeting the minimal contrast requirements defined in [[wcag2]] success criteria 1.4.3.

The other feature for enhanced display is largePrint which is used to indicate that a publication has been formatted for users who require larger font sizes to read.

<meta property="schema:accessibilityFeature">largePrint</meta>

There is no single accepted definition of what constitutes large print, so publishers should consult regional guidelines, when available, for what is acceptable. In general, though, a font size of 14pt to 16pt is considered enlarged print, while a font size of 18pt and above is considered large print. For this reason, it is recommended to include information about the large print formatting used in the accessibility summary so that users are not left guessing which size has been used.

In practice, large print is rarely used in EPUB publications. Reflowable EPUBs can typically have their font size, family, line spacing, and other properties adjusted to whatever the user prefers (see the displayTransformability feature). This is a much better feature for users than picking the specific formatting.

It is also problematic to indicate that reflowable EPUB publications are formatted for large print reading when user preference settings are typically applied to the text. The formatting the publisher has provided may not be viewable unless the user reverts their own preferences.

The largePrint feature must never be used to indicate that it is possible to reformat the text to a larger size (i.e., that the display is transformable). Features are set based on the defaults of a publication not on what it might be possible to achieve.

The most likely scenario for setting the largePrint feature is when a print-equivalent reproduction of a physical large print book is created using EPUB 3 fixed layouts. But the reality is that such works are rarely, if ever, created.

Image description

In this section:

  • alternativeText
  • longDescription

Providing text alternatives is key to making images accessible to readers who cannot perceive them or have difficulty processing the content.

There are two primary features that identify that accessible alternatives have been provided: alternative text and extended descriptions. Alternative text is a short description that can stand in for the image (usually added to the [[html]] [^img/alt^] attribute in XHTML content documents [[epub-3]]) while extended descriptions provide additional detail when images express information that cannot be conveyed through a brief description.

With the exception of decorative images — those that convey no information needed to understand the text — all images require alternative text to be accessible. The accessibility feature for indicating that alternative text has been provided is alternativeText.

<meta property="schema:accessibilityFeature">alternativeText</meta>

The alternative text provided for images must be meaningful both to meet accessibility requirements and to claim the alternativeText accessibility feature. A publication that simply auto-fills every [[html]] [^img/alt^] attribute with the text "Image", for example, is not providing users with a meaningful alternative.

Sometimes images will contain more information than can be expressed in a short alternative text field. In these cases, it is also necessary to provide an extended description. The accessibility feature for indicating that alternative text has been provided is longDescription.

<meta property="schema:accessibilityFeature">longDescription</meta>

Similar to the requirement for claiming that alternative text has been provided, the extended descriptions for images must fully capture all information being conveyed through the image. It is not required that every image have a description, though, only those that require more extensive detail.

The name "long description" was commonly used in the past for extended descriptions and was the nomenclature behind the now obsolete longdesc attribute in HTML. The name was a misnomer, however, as the descriptions provide additional detail and complement alternative text, they are not simply longer in length. That is why "extended description" is now the common name. Changing the schema.org vocabulary would only add development costs, so the old name is still used in it.

Math

In this section:

  • describedMath
  • latex
  • MathML

There are several ways to provide accessible math equations in EPUB publications but support in reading systems [[epub-3]] has historically been a barrier to the adoption of the most accessible methods. For example, although MathML [[mathml3]] would intuitively be the most accessible way to add math, it was not well supported in reading systems until recently. Its rendering in older reading systems relied on a JavaScript library that greatly slowed the presentation the more equations there are to render so MathML was often avoided.

Despite the past support issues, MathML markup is the optimal way to represent math content because it allows for intelligent interpretation and voice rendering by assistive technologies, allows users to explore the components of an equation, and even allows copying and pasting of equations into external applications. And with improved rendering in newer reading systems, publishers can be assured that their content will display as intended.

When MathML markup is included in a publication, the MathML accessibility feature is set in the package document [[epub-3]] metadata.

<meta property="schema:accessibilityFeature">MathML</meta>

Because of the history of poor support for MathML in reading systems, there is also a legacy requirement to identify all documents that contain MathML in the manifest [[epub-3]]. This was to allow reading systems to selectively turn on MathJAX rendering without slowing down the rendering of documents that had no math, for example, or to present a fallback document with math encoded as images. The presence of MathML could be inferred from this metadata, but most agents that process metadata will not try to infer information because the practice is unreliable. Always set the accessibility feature.

Although ideally all math equations would be represented as MathML, in practice this is often impractical to do. Basic math equations can often be represented using ASCII text without any loss of information, for example. When claiming that MathML is available as an accessible feature, however, all, or substantively all, equations that would be ambiguous in pure text form should be represented using MathML markup.

LaTeX is another formatting syntax for expressing math equations. Unlike MathML, which is XML-based, LaTeX is a typesetting language that uses a system of text escapes to format equations.

When LaTeX is included in an EPUB publication, it is indicated in accessibility metadata using the latex accessibility feature.

<meta property="schema:accessibilityFeature">latex</meta>

Unlike other formal technology names, like MathML and ARIA, the spelling "LaTeX" is not used to identify the feature in the schema.org vocabulary. It is strongly recommended to use the lowercase vocabulary spelling to avoid issues with processors that are case-sensitive.

Reading systems do not natively support the rendering of LaTeX equations the way they do MathML, but it is possible to add a JavaScript library like MathJAX to visually render the equations. Due to the lack of consistent support for scripting in EPUB, however, most publishers do not include LaTeX formatting directly in their publications. It is more commonly found in the source from which an EPUB is generated, and converted to an image for the EPUB, otherwise users without a scripting-capable reading system will have to view and read the raw LaTeX formatting. As a result, it is not common for the latex feature to be declared in a publication's metadata.

While not the most desirable from an accessibility perspective, it is also possible to provide accessible images of math equations if high quality alternative text and descriptions are provided. The presence of described images is indicated in the metadata using the describedMath accessibility feature.

<meta property="schema:accessibilityFeature">describedMath</meta>

The problem with describing math images is that human language is often not as precise as the equations it tries to describe and there is no way for the user to inspect the original equation to try and clarify ambiguities. For example, the description "the square root of x and y" could be interpreted as either taking the square root of x and adding that value to y, or adding x and y together and taking the square root of the result.

Images of math are still prevalent in EPUB, however, because improvements to MathML rendering are recent and there are many current and legacy reading systems that do not render MathML well.

When the describedMath property applies to an EPUB publication, the alternativeText and longDescription features are also set, depending on how the math is described.

Multimedia

In this section:

  • audioDescription
  • closedCaptions
  • highContrastAudio
  • openCaptions
  • signLanguage
  • transcript

Audio and video content often require a mix of features to improve their accessibility. While audio-only files only have to take their sound into consideration, video content typically mixes both important visual information with essential audio. But even a video without any sound may still convey information that has to be captured for users.

The practices for making audio accessible are generally the same regardless of whether the audio is part of a video or is a standalone clip. The biggest difference between the two cases is that audio with video typically requires more context to be accessible because the video content is also presenting important information.

One of the most common methods for making audio content in EPUBs accessible is by adding captions. There are two ways that this can be done: closed captions and open captions.

Closed captions are when the captions are defined separately from the audio content, allowing users to control their appearance and turn them on or off as they prefer. Their presence is indicated in the package document [[epub-3]] metadata using the closedCaptions accessibility feature.

<meta property="schema:accessibilityFeature">closedCaptions</meta>

Although closed captions are most beneficial to users who are deaf or hard of hearing, they can also be helpful to users who are blind or have low vision. It is possible for assistive technologies to read aloud the captions or render them as braille (although this is not common in EPUB reading systems [[epub-3]]). Audio descriptions and transcripts are generally more helpful for users who cannot see the content.

The typical way to add closed captions for audio in XHTML content documents [[epub-3]] is using the [^track^] element [[html]], with WebVTT [[webvtt]] being the most commonly used technology for captioning.

It is not possible to add closed captions natively to an audio-only clip in HTML. Although the [^audio^] element [[html]] supports tracks, unlike the [^video^] element it does not render any visual content. To work around this, the [^video^] element can be used to play audio-only clips with closed captions.

Open captions are the opposite of closed captions — they are a part of the visual display of a video. They cannot be turned off or have their appearance modified and cannot be accessed by assistive technologies. Their presence is indicated in the package document metadata using the openCaptions accessibility feature.

<meta property="schema:accessibilityFeature">openCaptions</meta>

Open captions are most beneficial to users who are deaf or hard of hearing; they do not make auditory content accessible to users who are blind. (While sign language interpretation is often preferable for users who are deaf, it is not easy to provide as will be discussed later in this section.)

Open captions cannot be used with audio-only clips in HTML unless the audio is rendered as a video (e.g., with a black background) in a [[html]] [^video^] element. The [[html]] [^audio^] element has no display.

While captions are helpful for those who cannot hear the audio, they are not always ideal. Many users who are deaf are more proficient at reading sign language than they are at reading written text. For these users, having an interpreter to translate what is being said on screen is preferable. The inclusion of sign language interpretation is indicated in the package document metadata using the signLanguage accessibility feature.

<meta property="schema:accessibilityFeature">signLanguage</meta>

Similar to captioning, there are two common ways to provide sign language interpretation. One is to include the interpreter as part of the video itself, often in a bubble in the lower corner of the screen. This method is best used where there is not a lot of action (e.g., for a person giving a speech) as it can obscure what is happening for all users.

The other method is to provide sign language interpretation in a separate video and link the two video elements so that they play back together. This method is problematic for ebooks both because HTML does not have a method of synchronizing two video elements (it would require scripting, which is not well supported in reading systems) and also because the author cannot know how the videos will be laid out and paginated (the sign language interpreter could end up on a separate page from the video). It is not common to find sign language interpretation in EPUBs for these reasons.

Another drawback of captions is that they only capture auditory content. For users who cannot see what is happening in a video, the context of what is occurring on screen is lost when only captions are provided. Audio descriptions fill this gap by describing what is happening visually (e.g. actor movements or expressions). Their presence is indicated in the package document metadata using the audioDescription accessibility feature.

<meta property="schema:accessibilityFeature">audioDescription</meta>

Another common method of making both the audio and video content accessible is through a transcript. A transcript, like an audio descriptions, can also describe more than just what is being said. For example, it can be structured more like a screenplay. For this reason, they are also useful for explaining what is happening in video-only content like animated instructional videos. When transcripts are provided, their presence is indicated in the package document metadata using the transcript accessibility feature.

<meta property="schema:accessibilityFeature">transcript</meta>

One drawback of using transcripts is that they are not available in real time like captions and audio descriptions. As a result, users who can see the content but not hear it may have to watch the video once in silence and then try to match up through memory what they are reading in the transcript against what they just watched. Even with detailed transcripts that include the actions to help orient the user, the time to process the video is greatly increased.

A final audio consideration for users who are hard of hearing or have difficult processing speech is to provide greater contrast between spoken content and background noises. High contrast audio refers to audio of speech that has no background noise, that maintains a difference of at least 20 decibels between the foreground speech and background noise, or that allows users to turn off any background noise. Its presence is indicated in the package document metadata using the highContrastAudio accessibility feature.

<meta property="schema:accessibilityFeature">highContrastAudio</meta>

For specific details on evaluating when auditory content meets the requirements to be classified as high contrast, refer to success criterion 1.4.7 [[wcag2]]. Not all audio has to meet this requirement, for example. The audio must be primarily speech and there is an exemption for musical content.

Information about how the audio meets the requirement should be provided in the accessibility summary (i.e., there is no background noise, at least 20db difference between foreground speech and background noise, or the background noise can be turned off.)

EPUB publications [[epub-3]] will often contain a mix of the above features to make audio and video content accessible. It is not required to pick only one feature for all audio and video clips, nor should publishers feel they have to limit themselves to providing only one accessible alternative per clip. As already discussed, different users will prefer different options.

Navigation

In this section:

  • ARIA
  • index
  • structuralNavigation
  • tableOfContents

EPUB requires that every publication provide a table of contents in the navigation document, so it is almost always possible to claim the tableOfContents accessibility feature.

<meta property="schema:accessibilityFeature">tableOfContents</meta>

The only time it would not be appropriate to claim tableOfContents is if the table of contents is incomplete — for example, if a publisher only links to one chapter to meet the absolute minimum requirements for a table of contents to be valid.

It is not required that the table of contents provide links to every sublevel heading in the publication. If the publisher wants to keep the table of contents in the navigation document simpler for reading system [[epub-3]] use, they can also provide a more detailed table of contents in a regular XHTML content document in the publication spine [[epub-3]]. (The tableOfContents value does not only apply to the table of contents in the navigation document, although this is the one most easily accessed by users.)

Providing users access into the major sections of the publication through the table of contents is one of EPUB 3's built-in methods of meeting the multiple ways success criterion [[wcag2]], the other being via the page-to-page navigation offered from the spine. Publishers should also consider adding indexes to make it easier for users to more rapidly locate specific information of interest. These can include general indexes, such as by subject or name, or more content-specific indexes, such as an index of recipes for a cookbook.

The presence of an index is indicated in the package document [[epub-3]] metadata using the index accessibility feature.

<meta property="schema:accessibilityFeature">index</meta>

The type of index provided does not matter when setting the index feature, but to set the feature the index must provide hyperlinks to the reference locations in the work or, if the index cites page numbers, must include a page list so that users can use it to navigate to the page references that way. If no method of accessing the reference locations is provided, the index feature must not be set.

Another method for users of assistive technologies to navigate a publication is through ARIA landmarks [[wai-aria]]. These are expressed using the [[html]] [^/role^] attribute and, particularly for EPUB, using the roles in the DPUB-ARIA extension [[dpub-aria]]. When an XHTML content document is loaded, assistive technologies typically check for the presence of landmark roles and provide users with a quick navigation menu.

The presence of ARIA landmarks is indicated in the package document metadata using the ARIA accessibility feature.

<meta property="schema:accessibilityFeature">ARIA</meta>

ARIA landmarks should be used sparingly as their usefulness diminishes the more there are in a document. On the web, they are used to direct users to the major components of a page's layout, such as the banner, search bar, navigation menu, main body, etc. A common mistake with publications is to replicate the entire table of contents as landmarks. This practice is not helpful as the hierarchical structure of the sections and subsections in the page is lost when using landmarks — the landmark links are always presented as a single flat list; there is no ability to nest levels like in a table of contents. A general rule of thumb for ARIA is that the usefulness of landmarks begins to diminish once you have more than a half-dozen or so in a document.

The final accessible navigation feature is the ability to move through the structure of a publication by heading level in the markup. When headings are properly marked up using the [[html]] h1 through h6 elements, it is possible for reading systems and assistive technologies to provide users hotkeys that allow them to move to the next heading.

The provision of structural navigation by heading is indicated in the package document metadata using the structuralNavigation accessibility feature.

<meta property="schema:accessibilityFeature">structuralNavigation</meta>

This feature is similar to having a table of contents but does not require the user to leave their current position and search the table of contents to find the next heading.

Publications that claim to provide structural navigation should ensure that the number of each heading element used matches the hierarchy of the heading within the publication. Although this is not required to conform to the info and relationships success criterion [[wcag2]], claiming structural navigation is more than just ensuring some heading tag has been used to identify each heading.

Some minor deviations from this rule are acceptable so long as the variations are consistent and users are informed of the change in structure in the accessibility summary or in front matter on how to use the book. For example, some publishers set a specific starting level for all their sidebars, such as <h3>, regardless of the heading level the sidebar falls under.

Pagination

In this section:

  • pageBreakMarkers
  • pageNavigation
  • a11y:pageBreakSource

Whether an EPUB publication [[epub-3]] is produced with a print equivalent or not, being able to locate and move to static page break locations is an important feature to help users navigate the content. It is useful in educational settings, for example, where users who are blind might use an EPUB to read an assignment instead of the print version their peers are using.

The first feature for identifying static pagination is static page break markers. These markers are inserted into the markup XHTML content documents [[epub-3]] using the doc-pagebreak role [[dpub-aria]] to make them readable by assistive technologies.

Static page break markers, as their name suggests, do not change depending on the device being used or the font and spacing preferences of the user. The pagination built into many reading systems [[epub-3]] is not static because it changes depending on these display factors.

The presence of static page break markers is indicated in the package document [[epub-3]] metadata using the pageBreakMarkers accessibility feature.

<meta property="schema:accessibilityFeature">pageBreakMarkers</meta>

The epub:type attribute [[epub-3]] value pagebreak [[epub-ssv]] is often paired with doc-pagebreak role for historical reasons but the value does not provide any accessibility benefits. The pageBreakMarkers feature must not be set if only the epub:type attribute is specified.

It is not required that every page break marker from the source be present, or that they be in the same order as the source, as digital publications often omit material that is only relevant to print and have front and back matter rearranged. The markers for blank pages at the end of chapters and sections are also often omitted from digital publications. When page markers are omitted, the accessibility summary should be used to explain why.

If the page breaks markers provided have limited utility, the pageBreakMarkers feature should not be claimed. For example, if only the page breaks at the start of chapters are identified, the ability to synchronize reading is no better than what the table of contents will provide. At a minimum, if the feature is claimed in cases like this, the limitation must be explained in the accessibility summary.

While static page break markers can be useful on their own, they are most commonly paired with a page list to allow users to quickly find a destination. The page list, as its name suggests, is a list of hyperlinks to each static page break location.

The presence of a page list is indicated in the package document metadata using the pageNavigation accessibility feature.

<meta property="schema:accessibilityFeature">pageNavigation</meta>

Similar to page break markers, a page list is most useful when it includes links to all the static page break locations. If the list is incomplete because blank pages or certain front or back matter is omitted, the summary should explain why. If the page list is mostly incomplete, such as when page breaks are only provided for chapter start locations, it is recommended not to claim it as an accessibility feature.

It is not required that a page list link to static page break markers identified by an [[html]] [^/role^] or epub:type attribute values. A destination ID for each page break location is all that is required.

When static page break markers, a page list, or both, are included, the source of the pagination needs to be identified, even when there is no source (publications with no print or other statically paginated version). As there may be multiple print editions of a work, all with different pagination, it is important for users to know which one the EPUB's pagination matches up with.

The source of the pagination is identified in a a11y:pageBreakSource property. It is recommended to use a unique identifier such as an ISBN or ISSN to identify the source, when available.

<meta property="a11y:pageBreakSource">
   urn:isbn:978765432100
</meta>

The pagination source is not an accessibility feature and there is no value to set for it using the schema:accessibilityFeature property.

If a unique identifier is not available, provide as much information as possible that can uniquely identify the source.

<meta property="a11y:pageBreakSource">
   Moby Dick; or, the Whale, Herman Melville, New York: Harper and Brothers, 1851. 
</meta>

If the pagination is unique to the digital publication (there is no static source), use the value "none".

<meta property="a11y:pageBreakSource">none</meta>

Portability

In this section:

  • unlocked

The absence of digital rights management (DRM) schemes is a critical accessibility feature as it ensures that users can access the content with their preferred reading systems [[epub-3]] and assistive technologies, which might otherwise be blocked by encryption. It also allows for personal use and transformations of the content that may be necessary for a user to perceive and understand it (e.g., using it with specialized text-to-speech software).

That a publication is not DRM protected is indicated in the package document [[epub-3]] metadata using the unlocked accessibility feature.

<meta property="schema:accessibilityFeature">unlocked</meta>

Do not set the unlocked feature unless it is known for certain that the publication will be available to users without DRM. If, for example, the publication will be sent to a distributor who will pass it on to multiple vendors, some of whom may apply DRM to the publication, the feature must not be set, even if it might be true in some cases.

The use of DRM does not mean that a publication will not be usable by assistive technologies. When DRM is applied, publishers should ensure that they do not add restrictions that could affect access, but even when this is done vendor DRM could still have impacts on the usability.

Reading order

In this section:

  • readingOrder

It is important to indicate if an EPUB has a defined reading order in the markup because it ensures that users of assistive technologies can follow the narrative in order. When the visual presentation of the content does not match the order in the markup, it makes it harder for non-visual readers to understand the text.

The presence of a reading order is indicated in the package document [[epub-3]] metadata using the readingOrder accessibility feature.

<meta property="schema:accessibilityFeature">readingOrder</meta>

Unlike many of the other accessibility features, where claims can be made even if coverage is not complete, the readingOrder feature must not be set unless the entire work conforms. Users cannot be expected to read the accessibility summary for problems with the reading order and then have to make sense of the mis-ordered content.

The lack of a logical reading order at the markup level most commonly occurs in fixed layout EPUBs. Reasons why include:

  • the fixed layout canvas makes it easy to use CSS to position elements in a sequence that is detached from the markup;
  • the nature of spreads (two pages side-by-side) allows publishers to create visual reading experiences that cross from one page to the next and back (for example, a sidebar that spans the bottom of a page).

Reflowable EPUBs typically do not have issues with their markup not reflecting the visual reading order, as there is less control over the layout position of content. But it is possible to create reading sequences that are out of order on a smaller scale than a page, as content can be positioned in a fixed manner within container elements. For example, a sidebar element could be given a fixed width and height and have elements positioned within that space.

Ruby annotations

In this section:

  • rubyAnnotations
  • fullRubyAnnotations

Ruby annotations are used as pronunciation guides for logographic characters in languages like Chinese, Japanese, and Korean (CJK). They make difficult CJK ideographic characters more accessible for language learners, native speakers of all reading levels, and people with reading disabilities. They also provide critical information for text-to-speech (TTS) engines.

Ruby annotations are implemented by attaching the [[html]] [^ruby^] tag to ideographic characters to provide the annotation. The [^rt^] tag contains the phonetic reading of the base character in the ruby tag.

Unlike other accessibility features, ruby annotations distinguish between publications that provide full coverage and partial coverage of all ideographic characters. This is because it is common for CJK publishers to provide different coverage options based on reader need. Publications with partial coverage target pronunciation support for rarer characters, proper nouns, or key terms without cluttering the text with ruby on every single character.

The presence of full ruby coverage, where all ideographic characters have annotations, is indicated in the package document [[epub-3]] metadata using the fullRubyAnnotations accessibility feature.

<meta property="schema:accessibilityFeature">fullRubyAnnotations</meta>

When not all CJK ideographic characters have ruby annotations, the rubyAnnotations value is used instead.

<meta property="schema:accessibilityFeature">rubyAnnotations</meta>

Scripting

In this section:

  • timingControl

The accessibility of scripted content is generally covered by conformance to WCAG 2 [[wcag2]]. As a result, the schema.org vocabulary currently only contains a single term related to timed content.

Timing is essential to much scripted interactivity. For example, a quiz will often have a time limit and interactive games often set a limit to complete activities. These time limits typically do not account for the additional time required for users with, for example, dexterity limitations or cognitive issues. For non-essential activities, it should be possible for users to control or turn off the timing so that they can complete the activity or better interact with it.

The ability to control the timing of scripted content is indicated in the package document [[epub-3]] metadata using the timingControl accessibility feature.

<meta property="schema:accessibilityFeature">timingControl</meta>

The timingControl feature is aligned with success criterion 2.2.1 Timing adjustable [[wcag2]]. Scripted content should meet the requirements of this success criterion to set the feature, by providing either the option to turn off timing or the ability to adjust or extend it.

Essential interactions, such as time limits on logins or for certain sessions to expire, that are exempted from conformance by the success criterion are also exempt for the purposes of setting this feature. However, these types of interactions are less common in EPUB publications.

Timing control is not limited to content with an explicit timer. The WCAG success criterion also consider text that scrolls across a banner or content that updates periodically as having a timed interaction component. It is important to check for these other cases even if they are not common in digital publications.

Tactile content

In this section:

  • braille
  • tactileGraphic
  • tactileObject

As EPUB publications [[epub-3]] can be read on refreshable braille devices, it is possible for a publication to include textual alternatives encoded using Unicode braille characters (see the Braille Patterns [[unicode]]).

The presence of braille encoded content is indicated in the package document [[epub-3]] metadata using the braille accessibility feature.

<meta property="schema:accessibilityFeature">braille</meta>

Do not use the braille feature for publications whose primary format is braille. The eBraille format, for example, is designed to represent braille character data in an EPUB-conformant package. It is not necessary to declare that an eBraille file has a braille feature since the format itself and its required metadata already make this fact clear. The braille feature is expected only when braille character data supplements the standard text.

Another advantage of the eBraille format is that it provides metadata to indicate the type of braille being included (language, code, grade, etc.). When braille is added to a standard EPUB, the accessibility summary is the only reliable way to provide this information to users, as distributors and reading systems [[epub-3]] are unlikely to check or report braille metadata.

Both eBraille and standard EPUB publications may contain tactile graphics. These are images that are specially designed to allow users who cannot see the images to explore them tactilely, either by printing the images (embossing) or rendering them on specialized refreshable braille devices.

The presence of tactile graphics is indicated in the package document metadata using the tactileGraphic accessibility feature.

<meta property="schema:accessibilityFeature">tactileObject</meta>

The tactile graphic does not have to be displayed in the content to claim this feature. If links are provided to external graphic files, the feature can still be claimed.

Refer to the BANA Guidelines and Standards for Tactile Graphics for more information about tactile graphic formats and formatting.

Although it may seem counterintuitive that a digital format like EPUB could contain a tactile object, the case is similar to external tactile graphics. If an EPUB provides links to files that can be downloaded and printed, it is possible to declare that tactile objects are available.

The presence of tactile objects is indicated in the package document metadata using the tactileObject accessibility feature.

<meta property="schema:accessibilityFeature">tactileObject</meta>

Word segmentation

In this section:

  • withAdditionalWordSegmentation
  • withoutAdditionalWordSegmentation

Word segmentation refers to whether additional spacing is added to languages that do not normally use whitespace characters to separate words (e.g., Chinese, Japanese, Thai, and Lao). Adding additional whitespace can make it easier to read for some users, especially those with cognitive or visual impairments who have difficulty following visually when there are no spaces.

The presence of additional word segmentation is indicated in the package document [[epub-3]] metadata using the withAdditionalWordSegmentation accessibility feature.

<meta property="schema:accessibilityFeature">
   withAdditionalWordSegmentation
</meta>

The lack of word segmentation is not an accessibility feature in the strictest sense, but it is included as a feature to indicate that the standard authoring practice for these languages — to not include extra space — has been followed.

The lack of additional word segmentation is indicated in the package document metadata using the withoutAdditionalWordSegmentation accessibility feature.

<meta property="schema:accessibilityFeature">
   withoutAdditionalWordSegmentation
</meta>

Setting one or the other of these values, as appropriate, is recommended to avoid ambiguity (i.e., leaving the user guessing if word segmentation is not provided or if the publisher forgot to add metadata). Do not, however, set either value for languages that normally use whitespace to separate words.

If a publication combines text with added word segmentation and text without, whichever method is predominant should be set as the accessibility feature. That the full text is not available in that mode should be explained in the accessibility summary.

Writing direction

In this section:

  • horizontalWriting
  • verticalWriting

Some languages, such as Chinese, Japanese, and Korean, can be written both vertically and horizontally, but the choice of writing direction can make it harder for some users to follow the text.

To indicate that horizontal writing is used, the package document [[epub-3]] metadata must indicate the horizontalWriting accessibility feature.

<meta property="schema:accessibilityFeature">
   horizontalWriting
</meta>

To indicate that vertical writing is used, the package document metadata must indicate the verticalWriting accessibility feature.

<meta property="schema:accessibilityFeature">
   verticalWriting
</meta>

Setting the appropriate writing direction is always recommended, but do not set either value for languages that are only written in one direction. For example, it adds no benefit to users to declare that English is written horizontally.

If a publication combines writing directions (for significant amounts of text, not for minor formatting like representing foreign language text), whichever method is predominant should be set as the accessibility feature. That the full text is not available in the specified direction should be explained in the accessibility summary.

Accessibility hazards

Setting hazards

There are three types of hazards that can affect readers of digital content:

  • flashing — if a resource flashes more than three times a second, it can cause seizures (e.g., videos and animations).

  • motion simulation — if a resource simulates motion, it can cause a user to become nauseated (e.g., a video game drawn on the [[html]] canvas element or parallax scrolling with CSS).

  • sound — certain sound patterns, such as ringing and buzzing, can cause seizures, while loud or sudden changes in volume can also negatively affect users.

It is not only important for users to be aware if these hazards are present but also if they are not present or if their status is unknown. Without this information, users would have no way of knowing if a publication is free from hazards or simply has not been checked, potentially leading them to obtain content which could be physically harmful to them.

To allow for complete metadata declarations, each of the hazards defined above can be in three different ways:

  • When hazards are present, the flashing, motionSimulation, and sound values can be declared.
  • When hazards are not present, the values to use are noFlashingHazard, noMotionSimulationHazard, and noSoundHazard.
  • When hazards are not known, the values to use are unknownFlashingHazard, unknownMotionSimulationHazard, and unknownSoundHazard.

Each hazard status is identified in the package document [[epub-3]] using a schema:accessibilityHazard property [[schema-org]]. The property is repeated for each status — do not group all the hazard statuses into a single tag.

<meta property="schema:accessibilityHazard">
   flashing
</meta>

<meta property="schema:accessibilityHazard">
   noMotionSimulationHazard
</meta>

<meta property="schema:accessibilityHazard">
   noSoundHazard
</meta>

These values can be paired together to identify each risk but authors have to ensure they do not accidentally declare conflicting statements for the same hazard type. For example, flashing, noFlashingHazard and unknownFlashingHazard must never be declared together in the same publication.

To help simplify authoring hazard metadata, there are also two global values that can be declared:

  • When there are no hazards of any type, the value none can be declared instead of setting a status for each individual hazard type.

    <meta property="schema:accessibilityHazard">
   none
</meta>

  • When none of the possible hazards is known, the value unknown can be declared instead of setting a status for each individual hazard type.

    <meta property="schema:accessibilityHazard">
   unknown
</meta>

These values are mutually exclusive and must never be declared for the same publication. Also, when either of these global values is declared, no individual hazard statuses should be declared as it is likely to cause conflicting statements and confuse users. For example, when none is declared, the flashing, motionSimulation, and sound hazards must not also be declared, nor any of their unknown value equivalents. Best practice is to never mix a global declaration with individual ones, even if they technically do not conflict.

It is not strictly an error to declare redundant statements. The none value could, for example, be paired with noFlashingHazard, noMotionSimulationHazard and/or noSoundHazard, since these statements are stating the same thing (none is equivalent to declaring the three individual statements). But it is strongly advised not to include redundant statements as they add no value and can lead to errors in the future if the hazard statuses have to change.

If an EPUB publication [[epub-3]] contains a hazard, or if the presence of a hazard type is not known, provide additional information in the accessibility summary. For hazards, this could include where the hazard occurs (both location within the publication and the point within the resource, when applicable). For unknown hazards, the reason why the hazard is not known should be stated, as well as whether and when a future update to the publication will provide more information.

EPUB Accessibility conformance

[[[epub-a11y]]] [[epub-a11y]] requires the status of all content hazards be identified using the accessibilityHazard property [[a11y-discov-vocab]].

The values unknown, unknownFlashingHazard, unknownMotionSimulationHazard, and unknownSoundHazard cannot be used to meet the reporting requirements for the property.

When hazards are known

In this section:

  • flashing
  • motionSimulation
  • sound

It is a critical health and safety issue to alert photosensitive users if a publication is known to contain content that flashes more than three times in any one second period or where the flash is below the general flash and red flash thresholds.

For the precise meaning of these definitions, refer to [[wcag2]] success criterion 2.3.1.

Content that can exhibit these characteristics includes videos, animated graphics, dynamic content drawn on the [[html]] [^canvas^] element, and CSS animations.

To indicate that there is a flashing hazard, the flashing hazard value must be declared in the package document [[epub-3]] metadata.

<meta property="schema:accessibilityHazard">flashing</meta>

Motion simulation is equally problematic as it can trigger severe dizziness, nausea, and disorientation in users with vestibular disorders.

Some examples of motion simulation include video games with a first-person perspective and CSS-controlled backgrounds that move at a different rate from the foreground content (parallax scrolling).

Refer to [[wcag2]] success criterion 2.3.3 for more information about motion simulation from animations. Although the success criterion makes an exception for essential motion simulation, that is only for conformance. A hazard should still be declared.

To indicate that there is a motion simulation hazard, the motionSimulation hazard value must be declared in the package document metadata.

<meta property="schema:accessibilityHazard">motionSimulation</meta>

What precisely constitutes a sound hazard, and how to test for these hazards, is not yet standardized. As a result, it is particularly important when declaring a sound hazard to explain what the hazard is in the accessibility summary.

To indicate that there is a sound hazard, the sound hazard value must be declared in the package document metadata.

<meta property="schema:accessibilityHazard">sound</meta>

When hazards are not present

In this section:

  • none
  • noFlashingHazard
  • noMotionSimulationHazard
  • noSoundHazard

Many books have no content that will pose a hazard to users. This is especially true of novels and similar books that consist only of headings and text. Instead of having to declare that each individual hazard type is not present, a shorthand to cover all hazards is provided in the none (hazard) hazard value.

<meta property="schema:accessibilityHazard">none</meta>

Setting the none value is the equivalent of setting the individual statements noSoundHazard, noMotionSimulationHazard, and noFlashingHazard.

If a publication consists of a mix of known, not applicable, and unknown hazards, then each hazard that is not present needs to be separately identified.

To indicate that there are no flashing hazards, for example, the noFlashingHazard value must be declared in the package document [[epub-3]] metadata.

<meta property="schema:accessibilityHazard">
   noFlashingHazard
</meta>

To indicate that there are no motion simulation hazards, the noMotionSimulationHazard hazard value must be declared.

<meta property="schema:accessibilityHazard">
   noMotionSimulationHazard
</meta>

And to indicate that there are no sound hazards, the noSoundHazard hazard value must be declared.

<meta property="schema:accessibilityHazard">
   noSoundHazard
</meta>

When hazards are not known

In this section:

  • unknown
  • unknownFlashingHazard
  • unknownMotionSimulationHazard
  • unknownSoundHazard

There may be times when the presence of accessibility hazards is not known but a statement is still needed in the package document [[epub-3]] metadata. This usually happens if a placeholder is needed until an accessibility evaluation is carried out.

To indicate that no hazards are known, the unknown (hazard) hazard value must be declared in the package document metadata.

<meta property="schema:accessibilityHazard">unknown</meta>

Setting the unknown hazard status is the equivalent of setting each of the individual unknown hazard statuses — unknownSoundHazard, unknownMotionSimulationHazard, and unknownFlashingHazard.

If the presence of hazards has not yet been determined, it is recommended to use the unknown value rather than the individual unknown statements for each type of hazard. It avoids the problem of missing one of the individual declarations and makes it easier to remove the statement once the hazards have been checked as there is only one metadata tag to delete.

If a publication consists of a mix of known, not applicable, and unknown hazards, then each hazard that is not known needs to be separately identified.

To indicate that the presence of flashing hazards is not known, for example, the unknownFlashingHazard value must be declared in the package document metadata.

<meta property="schema:accessibilityHazard">
   unknownFlashingHazard
</meta>

To indicate that motion simulation hazards are not known, the unknownMotionSimulationHazard hazard value must be declared.

<meta property="schema:accessibilityHazard">
   unknownMotionSimulationHazard
</meta>

And to indicate that the presence of sound hazards is not known, the unknownSoundHazard hazard value must be declared.

<meta property="schema:accessibilityHazard">
   unknownSoundHazard
</meta>

Accessibility summary

An accessibility summary provides a brief, human-readable description of the accessibility characteristics of an EPUB publication [[epub-3]] that cannot be expressed through the other discovery metadata. It is also used to provide additional context for accessibility feature claims.

The summary is provided using the schema:accessibilitySummary property [[schema-org]].

<meta property="schema:accessibilitySummary">
   MathML markup is only provided for complex equations.
   In all other cases, …
</meta>

The accessibility summary should not simply repeat the conformance information provided in the dcterms:conformsTo property [[dcterms]], for example, or the features listed in the schema:accessibilityFeature properties. When other accessibility metadata is present in the package document [[epub-3]], systems that process EPUB publications can already present it to users. Repeating it in the summary only makes them hear the information again.

<meta property="dcterms:conformsTo">
   EPUB Accessibility 1.1 - WCAG 2.1 Level AA
</meta>
<meta property="schema:accessibilitySummary">
   In addition to the requirements of its conformance claim,
   this publication includes sign language interpretation
   for all audio content.…
</meta>

If an EPUB publication does not meet the requirements for content accessibility in [[epub-a11y]], the reason(s) it fails should be noted in the summary. Similarly, if an EPUB creator is hesitant to make a formal claim of conformance, the reasons why can be explained in the summary.

<meta property="schema:accessibilitySummary">
   The publication is missing alternative text for complex diagrams.
   The publication otherwise meets WCAG 2.0 Level A.
</meta>
<meta property="schema:accessibilitySummary">
   Although this publication meets the requirements of its accessibility
   claim, the publication contains a motion hazard in chapter 8 page 202., 
   which could cause motion sickness in certain individuals. There is a video 
   showing a rollercoaster going up and down rapidly. Watching the video is 
   interesting, but not essential for understanding the concept of elevation 
   loss and gain.
</meta>
<meta property="schema:accessibilitySummary">
   This publication strives to meet accepted Web Content Accessibility 
   Guidelines (WCAG) at the AA level. Subject experts were used to create 
   the alternative text. A comprehensive index is included with links to 
   the top of the page.
</meta>
<meta property="schema:accessibilitySummary">
   AI was used to generate the alt text without review by people familiar 
   with the material.
</meta>
<meta property="schema:accessibilitySummary">
   People familiar with the material used AI to assist in the generation of Alt text.
</meta>

Do not repeat the schema:accessibilitySummary property to provide translations of a summary. EPUB does not define a method for including translations. Putting different xml:lang attributes on properties does not indicate a translation and could lead to wrong summary being rendered to users.

EPUB Accessibility conformance

[[[epub-a11y]]] [[epub-a11y]] only recommends an accessibilitySummary property [[a11y-discov-vocab]] be included if there is information to express that is not already captured by the other accessibility metadata.

Do not include an accessibility summary when there is nothing more to add to the conformance claim and other discovery metadata.

Overview

[[[epub-a11y]]] [[epub-a11y]] defines metadata for indicating whether a publication meets its conformance requirements, as well as how to report information such as who performed the evaluation, what credentials the evaluator holds, and who to contact for more information about the accessibility of a publication.

Although this metadata is not as descriptive as the schema.org properties, it is just as important to set. Reporting accessibility conformance to an accepted standard shows users that the content is broadly accessible without them having to go through all the specific features and access modes. The level of conformance, whether WCAG 2 Level A or AA, also provides insight into the user groups that will benefit from the accessibility affordances provided.

Reporting conformance also directly benefits publishers as it establishes trust with their customers. When customers see that a publisher consistently does the work to make their content accessible, they are more likely to seek out content from that publisher.

Reporting accessibility conformance is also important at a regulatory level. Many jurisdictions now have laws around the provision of accessible content. Indicating that a publication is conforming and providing details about the evaluation is often not optional metadata.

Conformance claims

For historical reasons, the EPUB accessibility vocabulary uses "certifier" and "certified" in the names of its properties. No official authority was intended from this choice of name, but it often causes confusion in regions where certification has legal meaning. The conformance metadata is only intended to describe an evaluator performing an evaluation of the content, which is why this document uses to the "evaluator" concepts to describe their uses.

Conformance statement

When an EPUB publication [[epub-3]] meets the discovery and accessibility requirements of [[[epub-a11y]]] [[epub-a11y]], a conformance claim can be made using the dcterms:conformsTo property [[dcterms]].

The EPUB Accessibility specification defines a controlled vocabulary of values [[epub-a11y]] that are allowed in the conformance statement. These values follow the pattern:

  1. the name and version of the EPUB Accessibility specification being conformed to (e.g., "EPUB Accessibility 1.2")
  2. a space-separated hyphen (" - ");
  3. the version and level of WCAG 2 that is met by the content (e.g., "WCAG 2.2 Level AA")
<meta property="dcterms:conformsTo">EPUB Accessibility 1.1 -  WCAG 2.0 Level AA</meta>
<meta property="dcterms:conformsTo">EPUB Accessibility 1.1 -  WCAG 2.1 Level AA</meta>
<meta property="dcterms:conformsTo">EPUB Accessibility 1.1 -  WCAG 2.2 Level AA</meta>
<meta property="dcterms:conformsTo">EPUB Accessibility 1.2 -  WCAG 2.2 Level A</meta>
<meta property="dcterms:conformsTo">EPUB Accessibility 1.2 -  WCAG 2.2 Level AA</meta>

These patterns only apply to EPUB Accessibility 1.1 [[epub-a11y-11]] specification and above. The EPUB Accessibility 1.0 [[epub-a11y-10]] specification was produced by a different organization and used URLs to identify conformance.

Conforming to the 1.0 standard is no longer recommended both due to its age and because it allowed conformance claims to WCAG 2 without fully meeting the requirements of that standard. Claiming conformance to EPUB Accessibility 1.0 makes it ambiguous whether the content is fully conforming which can prove problematic where accessibility is mandated by law.

It is important to ensure that an EPUB publication fully meets the requirements of the EPUB Accessibility specification and the version and level of WCAG specified. It is not sufficient, for example, to make a conformance claim based solely on an EPUB publication passing an automated conformance checker. While these tools are helpful for finding machine-identifiable issues, there are many checks that only a human can carry out.

There is currently no standardized way to state that a publication fails conformance reporting, as publishers typically omit any claims when a publication fails to meet minimum standards. One option is to use the word "none" in a dctems:conformsTo statement, but this might not be recognized as an accessibility claim if it is not accompanied by other evaluator information. It could also cause display issues in bookstores and reading systems [[epub-3]], as they will not expect a failing conformance statement and, for example, would not translate or modify the value to make more sense for users.

Multiple claims

It is possible for an EPUB publication to contain more than one conformance claim. For example, a publisher might add a dcterms:conformsTo tag that identifies an internal standard that the publication meets. Or a claim could be made to a non-accessibility standard.

It is possible for an EPUB publication to make multiple conformance claims against the EPUB Accessibility standard (e.g., to different versions), although this is strongly discouraged to prevent errors in display and confusion for users.

When an EPUB publication only contains one claim, there is no potential for misunderstanding any additional information provided, such as who performed the evaluation, the date of the evaluation, or where a detailed report can be found.

<metadata …>
   …
   <meta
       property="dcterms:conformsTo">
      EPUB Accessibility 1.2 - WCAG 2.2 Level AA
   </meta>
   <meta
       property="a11y:certifiedBy">
      John Doe Evaluation Services
   </meta>
   <meta
       property="a11y:certifierCredential">
      Acme Certified Evaluator
   </meta>
   <meta
       property="dcterms:date">
      2026-01-31
   </meta>
   <link
       rel="a11y:certifierReport"
       href="https://jdes.example.com/reports/a11y/moby-dick.html"/>
   …
</metadata>

But as soon as there are multiple claims, which of this additional metadata belongs to which claim can become ambiguous.

To make the links between the metadata clear, EPUB 3 includes the refines attribute [[epub-3]]. This attribute allows one piece of metadata (the child) to indicate what other piece of metadata (the parent) it is associated with by referencing the parent's ID in the refines attribute.

<metadata …>
   …
   <meta
       property="dcterms:conformsTo"
       id="conf">
      EPUB Accessibility 1.2 - WCAG 2.2 Level AA
   </meta>
   <meta
       property="a11y:certifiedBy"
       id="evaluator"
       refines="#conf">
      John Doe Evaluation Services
   </meta>
   <meta
       property="a11y:certifierCredential"
       refines="#evaluator">
      Acme Certified Evaluator
   </meta>
   <meta
       property="dcterms:date"
       refines="#evaluator">
      2026-01-31
   </meta>
   <link
       rel="a11y:certifierReport"
       refines="#evaluator"
       href="https://jdes.example.com/reports/a11y/moby-dick.html"/>
   …
</metadata>

The refines attribute can be used to link the metadata even if only one conformance claim is made but the choice to use it is optional. Not using the attribute is a choice that makes authoring the metadata simpler.

The examples in this guide show both methods — without a refines attribute and with one — to demonstrate how to mark up the information whichever case applies.

EPUB 2 does not have an equivalent to the refines attribute so there is no way to establish parent-child relationships. As a result, the metadata will always be ambiguous if it contains multiple conformance claims.

Adding an evaluator's name

Providing the name of the evaluator is required when a conformance claim is made. It is helpful to users in terms of assessing the quality of the evaluation performed. A user might be warier of a publisher self-certifying their work, for example, especially if they do not declare any evaluation credentials, than a trusted evaluation agency.

The name of the evaluator is provided in the package document [[epub-3]] metadata using the a11y:certifiedBy property [[epub-a11y]].

<meta property="a11y:certifiedBy">
   John Doe Evaluation Services
</meta>

When multiple conformance claims are made, the evaluator name is linked to the claim using the refines attribute [[epub-3]].

<meta property="dcterms:conformsTo"
      id="conformance">
   EPUB Accessibility 1.2 - WCAG 2.2 Level AA
</meta>

<meta property="a11y:certifiedBy"
      id="evaluator"
      refines="#conformance">
   John Doe Evaluation Services
</meta>

It is only necessary to add an ID to the meta tag for the evaluator if a credential, certification date, or the location of a detailed report is also going to be specified. Otherwise, it can be omitted.

If multiple parties performed an evaluation, it is possible to provide more than one a11y:certifiedBy property that links back to the conformance claim, but there is no guarantee if all, or even which one, will be displayed in a bookstore or reading system's [[epub-3]] metadata. As per the recommendation for other EPUB metadata, such as author names [[epub-3]], it is advised to put all the names in one tag if it is important that they all be displayed.

It is also possible to provide the evaluator's name in another language or script in EPUB 3 publications using the alternate-script property [[epub-3]], but this metadata is not often used.

Adding an evaluator's credential

To help users establish trust that the evaluator of the publication has the skills necessary to properly evaluate an EPUB publication [[epub-3]] for conformance, a credential can be attached to their name using the a11y:certifierCredential property [[epub-a11y]].

<meta property="a11y:certifiedBy">
   John Doe Evaluation Services
</meta>

<meta property="a11y:certifierCredential">
   Acme Certified Evaluator
</meta>

When multiple conformance claims are made, or there are multiple evaluators and the listed credentials do not apply to all of them, the refines attribute [[epub-3]] can be used to link each credential to its owner.

<meta id="evaluator" property="a11y:certifiedBy">
   John Doe Evaluation Services
</meta>

<meta property="a11y:certifierCredential" refines="#evaluator">
   Acme Certified Evaluator
</meta>

Ensure that the credential is not attached to the conformance statement, as it does not make sense that a conformance claim can have a credential.

There is no need for the credential meta tag to have an id attribute as no other metadata is attached to the credential.

The text to use for a credential is provided by the certifying agency.

Adding the evaluation date

Although users will typically assume that an evaluation was performed prior to the release of an EPUB publication [[epub-3]], it can be helpful to users to have the date explicitly stated. More often, though, this date is only used by the publisher to be able to determine when an evaluation was last performed.

The date of the evaluation is expressed using a dcterms:date property [[dcterms]]. The format of the data must be an [[iso8601-1]] conforming date string. The date is typically formatted as a four-digit year followed by a two-digit month and two-digit day, but the month and day are optional. For example, "2026-01-31", "2026-01", and "2026" are all valid date expressions.

<meta property="dcterms:date">
   2026-01-31
</meta>

More information about the recommended date format can be found in [[[datetime]]] [[datetime]].

When multiple conformance claims are made, the evaluation date is linked to the evaluator using the refines attribute [[epub-3]].

<meta id="evaluator" property="a11y:certifiedBy">
   John Doe Evaluation Services
</meta>

<meta refines="#evaluator" property="dcterms:date">
   2026-01-31
</meta>

When including an evaluation date, it is important to remember to update it if the publication is re-evaluated (e.g., for a revised release). Do not add a second dcterms:date property for the new date as it could lead to the wrong information being displayed to users.

Although the date format allows a time to be included after the date, in general there is little value in providing such detailed information. The time portion of the date string is also unlikely to be presented to users, so adding a time would only have value for internal processes.

Adding an evaluator's report

When performing a conformance evaluation, a detailed report is often produced to prove that a publication meets all the requirements of the EPUB Accessibility standard [[epub-a11y]] and WCAG success criteria [[wcag2]]. Making this report available can aid users who need more detailed information about the content and may be legally necessary to provide in order to sell a publication.

When a detailed report is available, it is possible to link to it from the package document [[epub-3]]. Unlike the other conformance metadata, however, the link is provided in a link element using the a11y:certifierReport property [[epub-a11y]].

<link rel="a11y:certifierReport"
      href="https://jdes.example.com/reports/a11y/moby-dick.html"/>

When multiple conformance claims are made, the report is linked to the evaluator using the refines attribute [[epub-3]].

<meta id="evaluator" property="a11y:certifiedBy">
   John Doe Evaluation Services
</meta>

<link refines="#evaluator" rel="a11y:certifierReport"
      href="https://jdes.example.com/reports/a11y/moby-dick.html"/>

Ensure that the report is hosted in a publicly readable location. Providing a link to an internal publisher document, for example, will prevent users from accessing it. The link should be stable so that the report remains accessible over time and to prevent security issues (e.g., to prevent broken-link hijacking when a domain expires and gets bought by a malicious party).

It is possible to include a copy of the report in the EPUB publication [[epub-3]] if it cannot be hosted on a secure, public server. Including reports in the EPUB container is not common practice, however, because it increases the difficulty to access it. Vendors and reading systems [[epub-3]] would have to extract the report and put it someplace accessible so that users can read it prior to buying or opening a publication. Otherwise, the only option is for the user to unzip the EPUB publication and locate the report themselves.

When embedding a report, the link element's href attribute would contain a relative path to the location of the file in the zip container. Because the report is not a publication resource, it does not have to be listed in the manifest [[epub-3]].

Conformance exemptions

In jurisdictions where accessibility conformance is mandated by law, it is also possible that some publishers are granted exemptions from full conformance. Exemptions may be granted, for example, for small and self-publishers who cannot afford the full costs of making their publications accessible or for content that cannot be made accessible without fundamentally altering its presentation.

The a11y:exemption property [[epub-a11y-exemption]] is used in these cases to indicate that an EPUB publication [[epub-3]] is exempt from a jurisdiction's accessibility requirements.

<meta property="a11y:exemption">
   eaa-microenterprise
</meta>

A publication can fail the minimum accessibility requirements for a jurisdiction while still meeting the minimum conformance requirements of the EPUB Accessibility standard. This is because conformance claims can be made when a publication meet WCAG 2 Level A while most legislations requires Level AA conformance. For this reason, the exemption is never linked to a conformance claim or an evaluator using the refines attribute [[epub-3]].

At the time of writing, the European Accessibility Act (EAA) is the only legislation known to provide these types of exemptions. Publications must meet the minimum accessibility requirements of the Act or fall under one of the following three exemptions in order to be distributed within the bloc of member countries:

  • disproportionate burden — the accessibility requirements of the Act shall apply only to the extent that compliance: … (b) does not result in the imposition of a disproportionate burden on the economic operators concerned. An exemption under this clause is declared using the value "eaa-disproportionate-burden".
  • fundamental alteration — the accessibility requirements of the Act shall apply only to the extent that compliance: (a) does not require a significant change in a product or service that results in the fundamental alteration of its basic nature. An exemption under this clause is declared using the value "eaa-fundamental-alteration".
  • microenterprise — the accessibility requirements of the Act do not apply to an enterprise which employs fewer than 10 persons and which has an annual turnover not exceeding EUR 2 million or an annual balance sheet total not exceeding EUR 2 million. An exemption under this clause is declared using the value "eaa-microenterprise".

Publishers should seek legal advice before claiming any exemption to EAA rules. Exemptions may also be time-limited and require a periodic reassessment to confirm that they still apply.

Adding contact info

Providing a contact email address for accessibility inquiries is another helpful way to establish trust with users. While conformance and discovery metadata provide a useful picture of the accessibility of a publication, it cannot cover every need. Before purchasing content, users may want to reach out to the publisher for more information.

A contact email is also a helpful way for publishers to gather feedback on the actual accessibility of their content. Sometimes inaccessible content will slip through quality assurance testing. And while user testing before release, is ideal, it is not always possible and cannot verify every user's needs are met.

The email address to use to contact the publisher is specified in an a11y:contactEmail property [[epub-a11y]].

<meta property="a11y:contactEmail">
   a11y@example.com
</meta>

The contact email is never linked to a conformance claim or evaluator using the refines attribute [[epub-3]] as it is general to the publication.

The contact email does not have to go to the publisher. If they use a third-party that is responsible for the accessibility of their products, that party's address can be used. The key point is that the email should go directly to someone who can handle accessibility requests. It should not be a general-purpose feedback address unless there is no other option.

Evaluating content readability metadata

Evaluation process

This section provides a process for evaluating which access modes and sufficient access modes to declare for an EPUB publication [[epub-3]].

The order in which the checks are performed is significant. Begin with the first check and only skip checks as instructed in the evaluation.

  1. Check 1: Does the publication contain text content?

    How to evaluate

    Verify if there is textual content encoded using Unicode characters. Images of text are not included in this check.

    In addition, any textual content that precedes or follows the work itself, such as copyright information, author biographies, lists of other works by the author or related works, promotional quotations, and similar is excluded from this check.

    For this check, recall that textual adaptations, such as alternative text and descriptions for images, or transcripts for audio, do not constitute textual content.

    • If the answer is no, skip to check 2.

    • If the answer is yes but the only content of the publication is a list of headings for audio synchronization, skip to check 6.

    • Otherwise, add a textual access mode declaration to the package document metadata:

      <meta property="schema:accessMode">textual</meta>

    Check 1.1: Does the publication only contain textual content?

    • If the answer is no, skip to check 2.

    • Otherwise, add a textual sufficient mode declaration to the package document metadata:

      <meta property="schema:accessModeSufficient">textual</meta>

      Skip to check 6.

    Explanation of check 1

    The first part of this check is to determine if a textual access mode applies. Most EPUB publications will have at least some text content. Only some fixed layout works, such as comics, manga, and children's books, are entirely image based.

    In the case there is no text content, or the work is not entirely textual, the evaluation continues at the next check for visual content.

    If the only text is the headings of the work — to provide support for structured audio playback via media overlays — there is no need to perform the subsequent checks for the other access modes. A textual access mode is not reported because in this case the work will be treated like an audiobook.

    If the work consists entirely of text content, such as is often the case with novels, there is no need to perform the additional access mode checks. A textual sufficient access mode can be declared and the next check jumps ahead to determine if there is audio synchronization.

  2. Check 2: Does the publication contain visual content?

    How to evaluate

    Check if there are graphics, photos, video, games, images or text, or other visual representations that carry information that a user must be able to access.

    Do not include the cover page image, publisher and imprint logos, author portraits, and purely decorative images in the determination. These are not essential visual content.

    • If the answer is no, skip to check 3.

    • Otherwise, add a visual access mode declaration to the package document metadata:

      <meta property="schema:accessMode">visual</meta>

    Check 2.1: Does the publication only contain visual content?

    • If the answer is no, skip to check 3.

    • Otherwise, add a visual sufficient mode declaration to the package document metadata:

      <meta property="schema:accessModeSufficient">visual</meta>

      Skip to check 5.

    Explanation of check 2

    The first part of this check is to determine if a visual access mode applies. Many EPUB publications will have at least some visual content that is essential to understanding the work.

    In the case there is no visual content, or the work is not entirely visual, the evaluation continues at the next check for auditory content.

    If the work consists entirely of visual content, such as is typically the case with comics and manga, there is no need to perform the additional access mode checks. A visual sufficient access mode can be declared and the next check jumps ahead to determine if there is audio synchronization.

  3. Check 3: Does the publication contain auditory content?

    How to evaluate

    Check for standalone audio clips, such as speeches or readings, or audio tracks for video that include information necessary to understand the work.

    Do no include audio that has no informational value, such as mood music or background music to an instructional video. Also, do not include audio synchronized via media overlays.

    • If the answer is no, skip to check 4.

    • Otherwise, add an auditory access mode declaration to the package document metadata:

      <meta property="schema:accessMode">auditory</meta>
    Explanation of check 3

    This check determines if an auditory access mode applies. Some EPUB publications contain auditory information, such as prerecorded speeches or as part of a video, but as EPUB is not an audiobook format it is generally never the case that an entire EPUB publication is authored as auditory content. That is why, unlike the first two checks, there is not an additional check that all the content is auditory.

    Note that audio synchronization is a special case and is handled later by the check for media overlays. An EPUB may be fully readable in auditory form via media overlays, but that is a different case from the primary authored content being auditory.

    In the case there is no auditory content the evaluation continues at the next check for tactile content.

  4. Check 4: Does the publication contain tactile content?

    How to evaluate

    Check if the text of the publication is authored using Unicode braille characters and/or if tactile graphics are embedded in the publication.

    Do not include adaptations like links to tactile graphics or objects, or braille text when it is only used as a text alternatives to visual content.

    • If the answer is no, skip to check 5.

    • Otherwise, add a tactile access mode declaration to the package document metadata:

      <meta property="schema:accessMode">tactile</meta>

    Check 4.1: Does the publication only contain tactile content?

    • If the answer is no, skip to check 5.

    • Otherwise, add a visual sufficient mode declaration to the package document metadata:

      <meta property="schema:accessModeSufficient">tactile</meta>

      Skip to check 6.

    Explanation of check 4

    The first part of this check is to determine if a tactile access mode applies. It is generally not common for mainstream EPUB publications to have tactile content as a primary means of conveying information (it is rare for them to have tactile content even as adaptations). Tactile content is more typically found in formats like eBraille.

    In the case where there is no tactile content, or the work is not entirely tactile, the evaluation continues at the next check for text alternatives.

    If the work consists entirely of tactile content, such as is the case with eBraille, there is no need to check for text alternatives. A tactile sufficient access mode can be declared and the next check jumps ahead to determine if there is audio synchronization.

  5. Check 5: Are there sufficient text alternatives for all non-text content?

    How to evaluate

    For this check, all the visual and auditory content needs to evaluated to verify that it includes textual adaptations that convey equivalent information. Images, for example, will need a combination of alternative text and extended descriptions to convey the same information in text form.

    • If the answer is no, skip to check 6.

    • Otherwise, add a textual sufficient access mode to the package document metadata:

      <meta property="schema:accessModeSufficient">textual</meta>
    Explanation of check 5

    This check is only reached for publications that contain a mix of more than one access mode. It is used to determine whether there is also a fully textual pathway to reading the content, making the publication useful for text-to-speech or dynamic braille translation.

    Note that this evaluation process does not consider the presence of single visual or tactile sufficient access modes for publications with a mix of primary access modes. Such single access modes are either not realistic, in the case of tactile, or extremely unlikely given the technological limitations of the format (e.g., alternative sign-language interpretation of an entire work).

  6. Check 6: Does the publication provide synchronized playback of prerecorded audio with the text?

    How to evaluate

    Check for the presence of media overlays [[epub-3]] that synchronize the text of the publication with prerecorded audio.

    The coverage of the overlays also needs to be verified — whether it covers the entire publication and whether all the equivalent information is available in auditory form.

    For example, if a media overlay for a children's picture book only provides audio for the words on a page and omits any description of accompanying image, the overlay does not provide an equivalent reading experience. The ability to perceive the image is still needed.

    • If the answer is no, skip to check 7.

    • Otherwise:

      • If the audio covers the entire work:

        • If the publication is structured audio (i.e., the only text content is the headings of the work), set both an auditory access mode and sufficient access mode in the package document metadata:

          <meta property="schema:accessMode">auditory</meta>
<meta property="schema:accessModeSufficient">auditory</meta>

        • Otherwise, set only an auditory sufficient access mode in the package document metadata:

          <meta property="schema:accessModeSufficient">auditory</meta>

      • In the case where the audio coverage is not complete, neither an auditory access mode nor a single auditory sufficient access mode is set.

        A sufficient access mode that combines the auditory content covered by the partial media overlay with the access mode(s) of the content not covered by the overlay could be declared in this case, but such a calculation has to be done manually.

    Explanation of check 6

    This check determines if the inclusion of synchronized audio via a media overlay represents an auditory access mode, a sufficient access mode, a combination of both, or neither.

    Media overlays are a special case in EPUB because they approximate an audiobook but are more like an adaptation in most mainstream production cases. When the audio fully captures the information of the work, whether the overlay represents an audiobook or an add-on audio experience is determined by the content the audio is synchronized to.

    For accessible republishers, EPUB is used as a delivery format for the audio and may only include the major headings of the work to allow structured navigation. In this case, both an auditory access mode and an auditory sufficient access mode are specified since the work can only be read in full auditorily.

    For most mainstream publications, the audio is a complement to the content produced by the author. In this case, it is treated as an adaptation and an auditory sufficient access mode is added to the access modes of the content. (EPUB cannot account for two formats in one, so because EPUB is not thought of as an audiobook format the audio is considered the accommodation.)

    In the final case, where the audio overlay is incomplete, it is not required to set a sufficient access mode that mixes the auditory content with the access modes of the content it does not cover. Users can be made aware that there is incomplete coverage by the lack of a single auditory sufficient access mode couple with the presence of the synchronizedAudioText feature.

  7. Check 7: Did checking the content result in more than one schema:accessMode property being added to the package document metadata?

    If the answer is yes, another set of sufficient access modes can be added that compiles all the declared access modes.

    Explanation of check 7

    For publications with a mix of content types, this final check adds a last sufficient access mode set that accounts for all the access modes that were identified in the first three checks.

    For example, if a publication has textual and visual access modes, a sufficient access mode set like the following would be added at this check:

    <meta property="schema:accessMode">textual,visual</meta>

Introduction to accessibility features

This appendix provides introductory information on what accessibility features are and how to determine if an EPUB publication [[epub-3]] contains them. It is only for intended for individuals unfamiliar with their markup or styling.

Finding features in the markup does not always mean the corresponding metadata applies to a publication. An [[html]] [^img^] tag, for example, might have an [^img/alt^] attribute but the actual alternative text could not be sufficient to identify the image. Always check the usage requirements for a property before declaring it.

Alternative text

Alternative text provides a brief description of an image.

For XHTML content documents [[epub-3]] this text is found in [[html]] [^img/alt^] attributes on non-decorative [^img^] tag.

<img src="images/hawk.jpg" 
     alt="A majestic hawk perches on a tree branch" />

For SVG content documents [[epub-3]], the alternative text is usually contained in the title element [[svg2]]. If there is a desc element [[svg2]], it typically contains the extended description.

<svg …>
   <title>A majestic hawk perches on a tree branch</title>
   …
</svg>

ARIA structural roles

ARIA structural roles provide additional context for assistive technologies about the type of content the user is reading.

ARIA roles are expressed in XHTML content documents [[epub-3]] using the [[html]] [^/role^] attribute:

<section id="c01" role="doc-chapter" aria-labelledby="c01-hd">
   <h2 id="c01-hd">A new beginning...</h2>
   …
</section>

Audio descriptions

Audio descriptions provide additional information about what is going on in a scene beyond the dialogue. For example, the movements of actors.

Audio descriptions are provided using technologies like WebVTT [[webvtt]]. The description files are then linked to their corresponding videos via [[html]] [^track^] elements with their [^track/kind^] attribute set to the value "descriptions".

<video controls="">
   <source src="chapter4-video.mp4" type="video/mp4"/>

   <track kind="descriptions"
         src="chapter4-video-audio-description.vtt"
         srclang="en"
         label="English Audio Description"/>

  <p>Your browser does not support the video tag.</p>
</video>

Closed captions

Closed captions capture character dialogue in text form for display on screen during video playback. The captions can be turned on or off.

Closed captions are provided using technologies like WebVTT [[webvtt]]. The caption files are then linked to their corresponding videos using [[html]] [^track^] elements with their [^track/kind^] attribute set to captions.

<video controls width="480" height="360">
  <source src="ch1-interview.mp4" type="video/mp4" />

  <track kind="captions"
          src="ch1-interview-captions-en.vtt"
          srclang="en"
          label="English Captions"
          default="" />

  <track kind="captions"
          src="ch1-interview-captions-es.vtt"
          srclang="es"
          label="Spanish Captions" />

  <p>Your reading system does not support the video element.</p>
</video>

Described math

Described math refers to images of math content that have been made accessible through a combination of alternative text and/or extended descriptions.

<img src="images/quadratic.jpg" 
     alt="Quadratic equation" aria-details="quad-desc"/>
<details id="quad-desc">
   <summary>Description</summary>
   <p>In the quadratic equation, the variable x is equal 
      to …</p>
</details>

Display transformability

Display transformability is an assertion that the content does not include any impediments that would prevent a user from being able to adjust the content display to their preferences. For example, to change the font family or enlarge the font size.

There is no simple example of display transformability as it is not a feature added to the content but an indication that no problematic styling is present. Inline [[html]] [^style^] elements are often an indicator that the content may interfere with user restyling, for example, but not always — it also depends on what kind of styling is applied inline.

Extended descriptions

Extended descriptions provide additional context for an image that cannot be expressed in shorter alternative text runs.

There are many ways to associate extended descriptions with visual content. Some common patterns include the presence of aria-describedby or aria-details attributes [[wai-aria]] on an [[html]] [^img^] tags, but these attributes may not always be set even when a description is available.

<figure>
  <img src="complex-chart.jpg"
       alt="Annual sales growth bar graph"
       aria-details="sales-long-description">

  <details id="sales-long-description">
    <summary>Detailed description of sales growth chart</summary>
    This bar graph illustrates the company's annual sales growth from 2019 to 2023.
    The vertical axis represents revenue in millions of dollars …
  </details>
</figure>

As descriptions are often stored in a separate file, it is also common to find a hyperlink after an image to its description. The description could also be contained in a figcaption element if the image is in a figure.

High contrast audio

High contrast audio refers to audio content — whether in a standalone audio file or as part of a video — that meets the audio contrast requirements of [[wcag2]] success criterion 1.4.7.

The use of high contrast audio is not a markup feature so it is not possible to provide an example of how it would be represented in the content. The presence of audio and video content can be checked by searching for the [[html]] [^audio^] and [^video^] elements, but their audio tracks will require testing.

High contrast display

High contrast display refers to content that meets the visual contrast requirements of [[wcag2]] success criterion 1.4.6.

The use of high contrast display is not a markup feature that can be easily discovered. It requires checking the contrast of all text — represented as character data or in images — and its background.

The following rule sets a contrast ratio of 21:1 with black text on a white background but it could be overridden by more specific styles.

body {
   color: black;
   background-color: white
}

As styles can be overridden by the CSS cascade, determining if high contrast display is really being used typically requires computing the final styling (only images are not affected). For example, more specific CSS rules will override less specific ones, and the order CSS style sheets are imported also affects the precedence of rules.

Indexes

Indexes make it easier for users to locate content of interest in a work. Common types of indexes in books include general subject indexes, name indexes, geographic indexes, and recipe indexes.

An index is usually tagged inside either an [[html]] [^nav^] or [^section^] element. When marked up accessibly these tags will have a [^/role^] attribute value of doc-index [[dpub-aria]].

<nav role="doc-index" aria-labelledby="index">
   <h1 id="index">Index</h1>
   <div>
       <p>
           <span>Artificial Intelligence</span>
           <a href="chapter2.xhtml#ai-def">12</a>
       </p>
       <p>
           <span>Machine Learning</span>
           <a href="chapter3.xhtml#ml-basics">45</a>
       </p>
       …
   </div>
</nav>

LaTeX formatting

LaTeX is a text-based formatting language that allows both mathematical and chemical equations to be expressed. It is not human readable in its raw form but is meant to be transformed — into images or MathML — for presentation.

Equations can be wrapped in any of the following depending on whether they are to be rendered inline or as block equations:

  • $…$
  • \(…\)
  • \[…\]
  • \begin{xzy}…\end{xyz}

In the last example, "xyz" is a placeholder for a specific latex formatting command, like "math" or "equation".

\[x = \frac{-b \pm \sqrt{b^2-4ac}}{2a}\]
$x = \frac{-b \pm \sqrt{b^2-4ac}}{2a}$

Large print

Although large print is not commonly found in EPUB publications, it is possible to set the font size to a large print standard such as 18pt (point).

If a book is formatted in large print it is usually possible to find this out from the publisher's description or metadata. Large print books are usually labeled and sold as such.

It may be possible to find the font size from the styling information, but this is often non-trivial to do. For example, it may require converting a relative unit size like em or rem (root em) to pt as pt is rarely used in CSS declarations (e.g., 1.5em is equal to 18pt).

body {
   font-size: 1.5em;
}

CSS rules can also be overridden by other rules, so without applying the full cascade of styles it is not possible to know if a rule is actually applied to all the text or not.

Opening XHTML content document [[epub-3]] in a browser and using the browser's markup inspection tools to review the calculated styles is the most reliable way to find font sizes. Reading systems do not typically provide this kind of information and may reformat the font size from what the author specified.

MathML markup

MathML [[mathml3]] is a markup language used to encode mathematical and chemical equations for display. The markup can be embedded directly in EPUB content document [[epub-3]].

<math xmlns="http://www.w3.org/2001/mathml">…</math>

Because EPUB is XML-based, the math tag could also have a prefix (e.g., m:math or mml:math). This kind of tagging is only going to be found in legacy content, however, as reading systems [[epub-3]] used to rely on the MathJAX application to render equations. Modern reading systems, like browsers, expect unprefixed elements as part of their support for HTML.

Note that it is not typically necessary to locate the actual MathML markup to find out if it is present in an EPUB 3 publication. The package document manifest [[epub-3]] will indicate if there is MathML in any EPUB content document [[epub-3]] as its presence has to be declared it in the properties attribute [[epub-3]].

<item id="c01" src="xhtml/chapter01.xhtml" … properties="mathml" />

If MathML is present but not declared, running EPUBCheck will also cause an error to be reported, indicating which files to look in for the markup.

Open captions

Open captions serve the same function as closed captions but part of the video stream instead of stored in a separate captioning file. Consequently, with open captions the user cannot change their appearance or even turn them off.

The use of open captions cannot be demonstrated by a markup example. The only way to determine if there are open captions is to watch each video and see if captions appear on screen. There will be no option in the video for the user to toggle the captions on or off.

To additionally verify that open captions are being used, the [[html]] [^video^] element for a clip with visible captions can be checked to ensure it does not have a child [^track^] element that references a caption file. The presence of a caption file would indicate closed captions are used.

Page break markers

Page break markers provide users the ability to locate their position relative to a pre-paginted equivalent edition, such as a hard or soft cover version of a book. They also provide link destinations for a page list.

Page break markers can either be visible or invisible elements in the content. They are identifiable in the markup by having a doc-pagebreak role [[dpub-aria]].

Hidden page break marker:

<span id="page_5"
      role="doc-pagebreak"
      aria-label="5"/>

Visible page break marker:

<div id="page_6" role="doc-pagebreak">6</div>

Page list

A page list is a list of links to static page break locations. It allows users, for example, to coordinate their reading with others using a print edition of the work.

The version of EPUB determines how the page list is expressed.

In an EPUB 3 publication, the page list is found in the EPUB navigation document [[epub-3]]. It will be contained an [[html]] [^nav^] element with its epub:type attribute [[epub-3]] value "page-list" [[epub-ssv]].

<nav epub:type="page-list" hidden="">
   <h2 id="pglist">Page list</h1>
   <ol>
      <li><a href="chapter1.xhtml#p01">1</a></li>
      <li><a href="chapter1.xhtml#p02">2</a></li>
      …
   </ol>
</nav>

In an EPUB 2 publication, the page list is expressed in the NCX document [[opf-201]]. It will be contained in a pageList element.

<ncx …>
    …
    <pageList>
        <navLabel>
            <text>Pages</text>
        </navLabel>
        <pageTarget type="normal" id="page01" value="1" playOrder="1">
            <navLabel>
                <text>1</text>
            </navLabel>
            <content src="chapter01.xhtml#p01"/>
        </pageTarget>
        <pageTarget type="normal" id="page02" value="2" playOrder="2">
            <navLabel>
                <text>2</text>
            </navLabel>
            <content src="chapter01.xhtml#p02"/>
        </pageTarget>
        …
    </pageList>
</ncx>

Reading order

The reading order establishes the narrative flow of the content. It must be represented logically in the markup, not just in how the content gets displayed. Fixed layout publications, for example, allow authors to position content visually differently from how it is ordered in the markup, which leads to an inaccessible reading experience for those using assistive technologies.

Determining if the reading order is logical is not a unique feature that can be shown with a simple markup example. It can only be established by checking how the content is ordered between the [[html]] [^body^] tags of each XHTML content document [[epub-3]], as well as across documents.

Ruby annotations

Ruby annotations are pronunciation guides for Chinese, Japanese, and Korean characters. They are either provided for all characters or only difficult or obscure ones.

Ruby annotations are encoded in the [[html]] [^ruby^] tag.

<ruby>東<rt>とう</rt></ruby><ruby>京<rt>きょう</rt></ruby>に<ruby>行<rt>い</rt></ruby>きます。

Sign language

Sign language interpretation makes spoken language accessible to individuals who are deaf or hard of hearing.

Sign language interpretation is most commonly part of the video stream when it is provided (e.g., the interpreter is overlaid in a corner). Less common is that sign language interpretation is provided as a separate synchronized video. In both cases, it can only be located by watching the video content. Like with open captions, there is nothing at the markup level that indicates the presence of interpretation.

Structural navigation

Structural navigation is the ability for users of assistive technologies to skip from one heading to the next without having to go through the table of contents every time. It is enabled by having all the headings marked up at their correct level.

HTML provides the h1 through h6 elements for marking up headings.

<h2>The reckoning</h2>

It is also possible that ARIA roles and properties [[wai-aria]] are used to identify headings.

<p role="heading" aria-level="2">The reckoning</p>

Synchronized audio and text

Synchronized audio and text is better known as read-aloud functionality in EPUB reading systems. It is when the user can opt to have an aural rendering of the content played back as the corresponding text is highlighted.

Text and audio synchronization is provided in EPUB 3 using the media overlays feature. To see if media overlays are included, check if the media-overlay attribute has been set on any items in the package document manifest [[epub-3]].

<?xml version="1.0" encoding="UTF-8"?>
<package xmlns="http://www.idpf.org/2007/opf" version="3.0" unique-identifier="pub-id">
   …
   <manifest>
      <item id="chap1" href="content.xhtml" media-type="application/xhtml+xml" media-overlay="chap1-overlay" />
      <item id="chap1-overlay" href="overlay.smil" media-type="application/smil+xml" />
      …
   </manifest>
   …
</package>

An EPUB with structured audio (i.e., full audio but minimal text) is not considered to have text and audio synchronization even though media overlays are present. Refer to for more information.

Table of contents

All EPUB publications require a table of contents with links into the major headings of the work.

In EPUB 3 publications, the table of contents is contained in the EPUB navigation document. Inside this file will be a [[html]] [^nav^] element with the epub:type attribute [[epub-3]] value "toc" [[epub-ssv]].

<nav epub:type="toc">
   <h2>Table of Contents</h2>
   <ol>
      <li>
         <a href="wasteland-content.html#ch1">I. THE BURIAL OF THE DEAD</a>
      </li>
      …
   </ol>
</nav>

The EPUB navigation document can be located in the package document manifest [[epub-3]]. It is identified by an item element with its properties attribute set to "nav".

<manifest>
   <item id="nav" href="nav.html" media-type="application/xhtml+xml" properties="nav"/>
   …
</manifest>

For EPUB 2, the table of contents is found in the NCX document [[opf-201]]. It is contained in the navMap element.

<ncx …>
   …
   <navMap>
      <navPoint id="ch1">
         <navLabel>
            <text>I. THE BURIAL OF THE DEAD</text>
         </navLabel>
         <content src="wasteland-content.xhtml#ch1"/>
      </navPoint>
      …
   </navMap>
   …
</ncx>

These are the table of contents used by reading systems but it is also possible for EPUB 2 and 3 publications to contain tables of contents within the body of the publication.

Tactile content

Tactile content includes braille text, tactile graphics, and tactile objects included in an EPUB publication to aid reading for users who are blind or have low vision.

Braille Unicode characters are not common in EPUBs but could be provided as alternative text for an image.

<img … alt="⠠⠉⠕⠇⠇⠁⠛⠑ ⠷ ⠊⠍⠁⠛⠑⠎ ⠷ ⠡⠝"/>

Tactile images may be included in EPUBs using the [[html]] [^img^} element but there is nothing that distinguishes them from any other image at the markup level. More common is that they are linked to after a standard image.

<figure role="figure" aria-label="Bar chart of monthly sales">
    <img src="images/sales-chart.svg" alt="Bar chart showing sales for January, February, and March." />

    <figcaption>
      <p>Monthly sales data for the first quarter.</p>
      <a href="tactile/sales-chart.brf" download>Download tactile graphic (BRF format)</a>
    </figcaption>
  </figure>

Tactile objects, due to their nature, are only linked out to.

<figure role="figure" aria-label="Illustration of a DNA double helix">
    <img src="images/dna-helix.jpg" alt="A 3D rendering of a DNA double helix showing its twisted ladder structure." />

    <figcaption>
      <p>Figure 1.2: The structure of a DNA molecule.</p>
      <a href="3d-models/dna-helix.stl" download>Download 3D model for printing (.stl format)</a>
    </figcaption>
  </figure>

Timing control

Timing control is the ability for users to extend the time needed to complete timed tasks such as completing embedded quizzes or playing interactive games.

Timing control mechanisms are typically encoded into the JavaScript code, so there is no universal method for what they look like or how they function.

XHTML content document with timed quiz.

<!DOCTYPE html>
<html xmlns="http://www.w3.org/1999/xhtml">
<head>
  <title>Timed Quiz</title>
  <link rel="stylesheet" href="quiz.css" type="text/css" />
  <script src="quiz.js" defer></script>
</head>
<body>
  <h1>Quiz Question</h1>
  <p>What is the capital of California?</p>
  <textarea id="quiz-answer" aria-label="Answer"></textarea>

  <div class="timer-controls">
    <div>Time Remaining: <span id="time-display" aria-live="polite">60</span> seconds</div>
    <button id="start-btn">Start Timer</button>
    <button id="add-time-btn">Add 30 Seconds</button>
    <button id="pause-btn">Pause Timer</button>
  </div>

  <!-- This empty element will be used for important, non-disruptive announcements -->
  <div id="quiz-status" role="alert" aria-live="assertive"></div>

</body>
</html>

JavaScript (quiz.js) with non-disruptive alerts:

document.addEventListener('DOMContentLoaded', () => {
  const timeDisplay = document.getElementById('time-display');
  const startBtn = document.getElementById('start-btn');
  const addTimeBtn = document.getElementById('add-time-btn');
  const pauseBtn = document.getElementById('pause-btn');
  const statusDiv = document.getElementById('quiz-status');

  let timeLeft = 60;
  let timerId = null;

  function updateTimer() {
    timeLeft--;
    timeDisplay.textContent = timeLeft;
    if (timeLeft <= 0) {
      clearInterval(timerId);
      // Announce "Time is up!" in the status div instead of a disruptive alert().
      statusDiv.textContent = 'Time is up!';
    }
  }

  startBtn.addEventListener('click', () => {
    if (!timerId && timeLeft > 0) {
      timerId = setInterval(updateTimer, 1000);
    }
  });

  addTimeBtn.addEventListener('click', () => {
    if (timeLeft > 0) {
      timeLeft += 30;
      timeDisplay.textContent = timeLeft;
    }
  });

  pauseBtn.addEventListener('click', () => {
    if (timerId) {
      clearInterval(timerId);
      timerId = null;
      pauseBtn.textContent = 'Resume Timer';
    } else {
      if (timeLeft > 0) {
        timerId = setInterval(updateTimer, 1000);
        pauseBtn.textContent = 'Pause Timer';
      }
    }
  });
});

Text-to-speech enhancements

Text-to-speech enhancements are used to improve the quality of synthetically generated audio in reading systems. There are three technologies that can be used: SSML, pronunciation lexicons, and CSS Speech. Support for these technologies is extremely limited, however, so finding them in EPUB publications is rare.

SSML is markup that can be used inline in XHTML content documents [[epub-3]] to provide a pronunciation along with the phonetic alphabet used. It is represented using the ssml:ph or ssml:alphabet attributes [[epub-tts]].

<p ssml:alphabet="ipa"> … situated between <span ssml:ph="ˈθɜrti dɪˈgriz">30°</span>
    <span ssml:ph="ˈθɜrtiˈwʌn ˈmɪnɪtz">31′</span> 
    <span ssml:ph="ˈθɜrtiˈnaɪn ˈsɛkəndz">39″</span>
    … </p>

PLS lexicons contain dictionaries of terms and their pronunciations.

<lexicon version="1.0"
         alphabet="ipa"
         xml:lang="en"
         xmlns="http://www.w3.org/2005/01/pronunciation-lexicon">
  <lexeme>
    <grapheme>Bethesda</grapheme>
    <phoneme>bəˈθɛzdə</phoneme>
  </lexeme>
  <lexeme>
    <grapheme>Chattanooga</grapheme>
    <phoneme>ˌtʃætəˈnugə</phoneme>
  </lexeme>
  <lexeme>
    <grapheme>Jefferson</grapheme>
    <phoneme>ˈdʒɛfərsən</phoneme>
  </lexeme>
  …
</lexicon>

These files can be located in package document [[epub-3]] by searching for the media type application/pls+xml. XHTML content documents can also be checked for a link element declaring the lexicon.

Package document manifest declaration:

<manifest>
   …
   <item id="pls" src="lexicon.pls" media-type="application/pls+xml"/>
   …
</manifest>

XHTML content document declaration:

<html …>
   <head>
      …
      <link rel="pronunciation" type="application/pls+xml" href="lexicon.pls"/>
   </head>
   …
</html>

CSS Speech properties are set in style declarations. They will typically appear in separate CSS style sheet files, but could appear in style elements or even in inline style attributes (although this latter practice is generally discouraged).

body {
   voice-family: young female;
   voice-rate: normal;
}

section {
   pause: 0 20ms;
}

Transcripts

Transcripts provide a text account of audio content for users who cannot hear the content.

Transcripts are not usually uniquely identifiable in the markup beyond having a heading to indicate that what follows is a transcript. The transcript can be referenced from an aria-details [[wai-aria]] attribute on an [[html]] [^audio^] or [^video^] element, but the use of the attribute for this purpose is not widespread.

<audio controls 
       aria-details="audio-transcript"
       aria-label="Audio recording of the interview">
       …
</audio>

<details id="audio-transcript">
    <summary>View Transcript</summary>

    <div>
        <p>This is a transcript of the audio recording.</p>
        <p>[Interviewer] Welcome to the show.</p>
        <p>[Guest] Thank you for having me.</p>
    </div>
</details>

Unlocked content

Unlocked content refers to an EPUB publication that does not make use of digital rights management schemes.

If an EPUB publication [[epub-3]] has digital rights management applied to it, the rights.xml and/or encryption.xml files will be included under the META-INF directory.

But these files are typically added by the distributor/vendor, so it may not be possible to determine if a publication is unlocked or not without knowing how it will be distributed.

Writing mode

The writing mode refers to whether the text content is presented horizontally or vertically. For accessibility purposes, the distinction only matters for languages that can be written both ways (e.g., the Chinese, Japanese and Korean languages).

The writing mode of the text is declared in CSS writing-mode property declarations.

Vertical left-to-right layout

body {
   writing-mode: vertical-lr
}

Vertical right-to-left layout

body {
   writing-mode: vertical-rl
}

Horizontal top-to-bottom layout

body {
   writing-mode: vertical-lr
}

Checking raw CSS styles declarations is not reliable, however, as they may get overridden or not used during the cascade process. A publication will need to be reviewed in a reading system or browser that supports both horizontal and vertical writing to see which way the text gets laid out.

Word segmentation

Word segmentation refers to the use of space to separate written words. Not all languages use space to separate words, as is the case with Chinese, Japanese, Lao, Khmer, Burmese, and Tibetan languages, among others.

This method of writing makes it difficult for some readers to process the text, so publishers will sometimes provide additional spacing that can be toggled on or off.

The markup may indicate if additional word segmentation is provided (e.g., if there are span elements around each group of characters to separate), but this kind of check is not reliable unless the CSS styling to add space is also easily verifiable.

<p class="segmented-text">
   <span>東京</span><span>に</span><span>行きます</span>。
</p>

CSS styling to add a margin on the right:

.segmented-text > span {
  margin-right: 0.5em;
}

A more reliable way to check for word segmentation is to open a publication in a reading system and see if there is spacing between words.

Obsolete metadata

Obsolete properties

The following schema.org properties [[schema-org]] are no longer recommended for use.

schema:accessibilityAPI

The creator of the content is not responsible for the interaction between reading systems [[epub-3]] and the underlying platform APIs.

schema:accessibilityControl

This property does not differentiate issues arising from the reading system interface from those in the underlying content, which has led to confusion about its use.

Meeting the requirements of [[wcag2]] will mitigate most known issues with the content and is sufficient for authoring purposes.

Obsolete accessibility features

The following accessibility features from the [[[a11y-discov-vocab]]] [[a11y-discov-vocab]] are no longer recommended for use.

annotations

The use of the annotations value is now deprecated due to the general nature of annotations in published works. When authors or publishers add annotations to a work, they provide information for all readers; they are not used as a means of enhancing otherwise inaccessible content.

bookmarks

The use of the bookmarks value is now deprecated due to its ambiguity. Reading systems [[epub-3]] typically provide bookmark support not the publication itself.

captions

The use of the captions value is now deprecated. Authors should use the more specific closedCaptions or openCaptions values, as appropriate.

printPageNumbers

The metadata term printPageNumbers value has been replaced by pageBreakMarkers.

Inapplicable features

The following accessibility features from the [[[a11y-discov-vocab]]] [[a11y-discov-vocab]] do not apply to EPUB publications.

ChemML

There is no native support for rendering Chemical Markup Language (ChemML or CML) in HTML. Unlike MathML, it is not a part of the HTML standard so the markup cannot be embedded directly in XHTML content documents. It is also not a core media type in EPUB so any use of ChemML files requires a manifest fallback to a supported image. For these reasons, the ChemML term is not listed as a usable feature in EPUB metadata at this time.
